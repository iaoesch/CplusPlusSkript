\chapter{Zu diesem Skript}
Dieses Skript ist eine Einführung in die Objekt Orientierte Programmiersprache \Cpp.
In diesem Kurs geht es um das Kennenlernen einer objektorientierten Sprache und das Umsetzen eines OO Designs in ein lauffähiges Programm. Dabei wird \Cpp als Begleitsprache benutzt. Leider ist der Sprachumfang von \Cpp viel zu gross, um hier vollständig abgedeckt zu werden, deshalb wird das Schwergewicht auf die für OOP absolut notwendigen Gebiete gelegt. Unbehandelt bleibt insbesondere die Standard-Bibliothek.

\section{Grundsätzliches}

In C und \Cpp werden alle Statements (Befehle) mit einem Semikolon ';' abgeschlossen.
Variablen welche ausserhalb von Funktionen definiert werden, und Funktionen selbst sind grundsätzlich global, das heisst sie können von anderen Modulen benutzt oder aufgerufen werden. Um den Gültigkeitsbereich einer Funktion oder einer globalen Variable auf
ein Modul zu beschränken, muss diese als static deklariert werden, dann kann nur noch innerhalb des entsprechenden Moduls auf sie zugegriffen werden. Definitionen von Funktionen innerhalb von Funktionen (lokale Funktionen) sind nicht möglich.  

\section{Module}
Ein C/\Cpp Programm ist normalerweise in mehrere Module (Dateien) aufgeteilt. Ein Modul besteht aus einem \OviIndex{Deklarationteil} (\OviIndex{Schnittstellenbeschreibung}) in einer \OviIndex{Headerdatei}, normalerweise mit der Dateiendung .h oder .hpp, und einem \OviIndex{Implementationsteil} in einer Datei mit der Endung .cpp (Gelegentlich auch .c++ oder .cxx).

In der Headerdatei sind alle \OviIndex{Deklarationen} von Funktionen, Variablen, Konstanten, Datentypen und Klassen enthalten, die vom Modul zur Verfügung gestellt werden. Im Implementationsteil sind die entsprechenden \OviIndex{Definitionen} (Code und Speicherreservationen) enthalten, plus eventuell weitere lokale Variablen, Funktionen und Hilfsklassen.

In \Cpp wird üblicherweise jede Klasse in einem eigenen Modul implementiert, eventuell zusammen mit Hilfsklassen, die nur von diesem Modul verwendet werden.
Wenn ein Modul Dienste von einem anderen Modul in Anspruch nimmt, muss es dessen Headerdatei mit dem \#include-Befehl einbinden.

Jedes Modul (.cpp) wird vom Compiler selbständig und unabhängig von anderen Modulen (.cpp) Compiliert. Aus jedem Modul wird somit eine Objektdatei (Maschinencodedatei) erzeugt, welche erst vom Linker zum lauffähigen Programm zusammengefügt werden.

Fehlende Funktionen oder globale Variablen, oder doppelt definierte Funktionen werden deshalb meist erst vom Linker erkannt, und mit nicht sehr aufschlussreichen Fehlermeldungen geahndet ('Symbol xyz not found' oder 'Symbol xyz allready defined in Module qwert.cpp').

\section{Kommentare}
\Cpp kennt zwei Arten von Kommentaren: Den Zeilenkommentar und den Blockkommentar. Ein Zeilenkommentar beginnt mit der Zeichenfolge //, der Rest der Zeile gilt dann als Kommentar. Ein Kommentarblock wird mit /* eingeleitet und endet mit */. Ein Blockkommentar kann sich über mehrere Zeilen erstrecken. Blockkommentare können nicht verschachtelt werden.
\begin{CCode}
// Ein Zeilenkommentar
/* dieser Kommentar
   geht ueber mehrere
   Zeilen */
\end{CCode}

\section{Präprozessorbefehle}
Präprozessorbefehle erkennt man an dem vorangestellten \#. Dabei handelt es sich nicht um C oder \Cpp Befehle, sondern um spezielle Präprozessorbefehle. Diese Befehle dienen zum Verändern und Ersetzen von Text und dem bedingten Compilieren. Nachfolgend einige oft verwendete Präprozessorbefehle.

\begin{CCode}
#include "test.h"   // der wohl wichtigste Befehl, fuegt die angegebene Datei an 
                    // dieser Stelle in den Code ein.
#if 0               // Bedingte Kompilierung
...                 // Der Teil dessen Bedingung nicht zutrifft wird als
#else               // Kommentar behandelt, also vom Compiler ignoriert
...
#endif

#define pi 3.14     // Makro (definition einer Konstanten)

#ifdef DEBUG        // Bedingte Kompilierung
... // Wird nur compiliert wenn Makro Debug definiert ist
#endif
\end{CCode}
Speziell beim Einbinden von Headerdateien kann es geschehen, dass die selbe Datei über verschiedene Wege mehrfach eingebunden wird. Das kann zu Problemen führen, besonders bei Klassen, weil diese nur einmal definiert werden dürfen. Das Problem lässt sich umgehen, wenn der Inhalt jeder Headerdatei mit folgenden Befehlen eingeschlossen wird:
\begin{CCode}

#ifndef HEADER_DATEI_NAME_HPP // ifndef -> if not defined, Gegenteil von ifdef
#define HEADER_DATEI_NAME_HPP

// Inhalt der Headerdatei,    wird nur Compiliert wenn HEADER_DATEI_NAME_HPP noch
                           // nicht definiert war, also nur beim ersten Mal
#endif
\end{CCode}

Damit wird garantiert, dass der Inhalt der Headerdatei nur einmal berücksichtigt wird. Diese Methode ist beinahe Standard, man findet sie bei allen Systemdateien und Bibliotheken, und die meisten Programmierer wenden sie grundsätzlich an, um eventuellen Problemen vorzubeugen.
Es gibt zwei Formen für den Includebefehl:
\#include <......> sucht die Datei im Systemverzeichnis des Compilers und allfälligen weiteren vom Benutzer angegebenen Verzeichnissen. Diese Form wird für Bibliotheksfunktionen verwendet.
\#include ''.....'' sucht zuerst im gleichen Verzeichnis in welchem die cpp Datei liegt, und dann in denselben Verzeichnissen wie \#include <...>. Diese Form wird für Projektbezogene Dateien (Vom Benutzer selbst geschriebene Dateien) eingesetzt. 

Zusätzlich gibt es einige vordefinierte Makros, welche Informationen über den Compiler und Dateien enthalten: \_\_FILE\_\_, \_\_LINE\_\_, \_\_DATE\_\_, \_\_TIME\_\_, \_\_cplusplus und weitere. \_\_cplusplus ist definiert, wenn ein C++ Compiler compiliert.

\section{Blöcke}
Mit den geschweiften Klammern { } werden in C/\Cpp mehrere Anweisungen zu einem Block zusammengefasst. Variablen, die innerhalb eines Blockes definiert werden, hören auf zu existieren wenn der Block verlassen wird, sie haben nur innerhalb des Blockes ab der Definition Gültigkeit.

{
  int a, b, c;
  cout << "Bitte 2 Zahlen eingeben" << endl;
  cin >> a;
  cin >> b;
  c = a + b;
  cout << a << " + " << b << " = " << c << endl;
} // Variablen a b c ab hier nicht mehr gueltig
\end{CCode}
 
\section{Dynamische Speicherverwaltung}
Mit new kann dynamisch Speicher reserviert werden, und mit delete wieder freigegeben werden. Zur Freigabe von Feldern (Arrays) muss delete[] verwendet werden. Jeder mit new allozierte Speicherblock muss mit delete wieder freigegeben werden, sonst entstehen Speicherlecks. 

\begin{CCode}
int *Memory;            // Ein Zeiger auf int definieren

Memory = new int[200];  // Speicher fuer ein Array von 200 ints reservieren und 
                        // Zeiger darauf zeigen lassen

Memory[4] = 17;         // Ein Element im Feld beschreiben

delete[] Memory;        // Speicher wieder freigeben

Memory = new int;       // Platz fuer einen einzelnen int reservieren und 
                        // Zeiger darauf zeigen lassen

*Memory = 55;           // den int veraendern

delete Memory;          // und Speicher wieder freigeben

\end{CCode}

\section{Datei Ein- und Ausgabe}
\OviIndex{Dateiausgaben} laufen über Objekte vom Typ \OviIndex{ofstream}, Eingaben über solche vom Typ \OviIndex{ifstream}. Um Dateiobjekte erzeugen zu können, muss das Headerfile <fstream> mittels \#include <fstream> eingebunden werden. (Mit \#include werden dem Code Funktionen aus Bibliotheken zur Verfügung gestellt). Dateiobjekte können anschliessend wie cin und cout benutzt werden. (cin und cout sind eigentlich nichts anderes als zwei vordefinierte Dateiobjekte)

Nach der Erzeugung eines \OviIndex{Streamobjektes} sollte man mit if(Objekt) überprüfen, ob die Datei (Stream) erfolgreich geöffnet werden konnte. Das Objekt liefert im Fehlerfall false zurück. Die Methode eof() liefert true zurück, wenn das Dateiende erreicht ist (Oder ein Fehler vorliegt). Die Methode good() liefert true zurück, wenn die nächste Streamoperation klappen könnte, wenn sie false liefert, geht die nächste Operation bestimmt schief.

\begin{CCode}
#include <fstream>

using namespace std;

int main(int argc, char *argv[]) 
{
  int a = 4, b = 5;
  ofstream AusgabeDatei("c:\\test.txt"); // Dateiobjekt erzeugen
  if (AusgabeDatei) {
    AusgabeDatei << "Mein erstes File" << endl;
    AusgabeDatei << "a + b ist " << (a + b) << endl;
  } else {
    cout << "Fehler, konnte Datei nicht oeffnen" << endl;
  }
  return 0
} 
\end{CCode}



Die Art der Datei kann durch weitere Argumente beim Konstruktor näher spezifizert werden:
Einige dieser Argumente sind vom Standard definiert, andere sind Betriebssystem und Compilerabhängig. Mehrere Modi können mit | (binäres or) kombiniert werden.
ios\_base::app öffnet eine Datei zum Anhängen, der default ist überschreiben.
ios\_base::binary öffnet eine Datei im Binärmodus, der Default ist Textmodus.

Die Unterscheidung zwischen Binär und Textmodus ist vor allem unter Windows und Dos wichtig, hier wird im Textmodus das Zeichen 0x0A in die Zeichenkombination 0x0A 0x0D umgewandelt, was bei binären Daten fatal sein kann.
\begin{CCode}

  ofstream AusgabeDatei("c:\\test.txt", ios_base::app);    // Dateiobjekt zum 
                                                          // anhaengen erzeugen

  ofstream AusgabeDatei("c:\\test.txt", ios_base::binary); // Dateiobjekt im Binaer-
                                                          // modus erzeugen

  ofstream AusgabeDatei("c:\\test.txt", ios_base::binary|ios_base::app); 
                                                          // Binaere Datei zum 
                                                          // anhaengen erzeugen

  ifstream EingabeDatei("c:\\test.txt");                   // Dateiobjekt im Text-
                                                          // modus zum Lesen oeffnen

  ifstream EingabeDatei("c:\\test.txt", ios_base::binary); // Dateiobjekt im Binaer-
                                                          // modus zum Lesen oeffnen

\end{CCode}

Zur Behandlung von einzelnen Zeichen oder ganzen Zeilen stellen die Streamobjekte
die Methoden put(), write(), get(), getline(), read() und ignore() zur Verfügung. Diese Methoden sind
zum Teil mehrfach überladen und können sehr flexibel eingesetzt werden, besonders
wenn die Operatoren $<<$ und $>>$ nicht die gewünschte Funktionalität liefern. 

\subsection{Formatierte Ausgabe}
\index{Formatierte Ausgabe} Die Formatierung der Ausgabe wird über Flags gesteuert. Mit der
Methode Flags() können die Flags gesetzt oder gelesen werden, mit
setf können Flags hinzugefügt werden.

Es sind unter anderem folgende Flags definiert:

Wahl des Zahlensystems: ios\_base::dec, ios\_base::hex, ios\_base::oct 

Bündigkeit: ios\_base::left, ios\_base::right

Gleitkommaformat: ios\_base::scientific, ios\_base::fixed

 
Beispiel:
\begin{CCode}


  ios_base::fmtflags AlteFlags;
  
  AlteFlags = cout.flags(); // Flag Zustand lesen

  // Ausgabeformat waehlen (Flags zusaetzlich setzen)
  cout.setf(ios_base::left|ios_base::oct|ios_base::fixed);
  
  cout << 1.23; // Ausgabe im gewaehlten Format
  
  // Alle Flags loeschen, nur Hex ist gesetzt
  cout.flags(ios_base::hex);
  
  cout << 47;
  

\end{CCode} 

Flags bleiben solange aktiv, bis sie wieder explizit rückgesetzt werden. Mit der Methode 
precision kann die Genauigkeit (Anzahl signifikanter Stellen) von Fliesskommawerten
definiert werden, mit width() die Feldbreite und mit fill() das Füllzeichen gewählt werden:
\begin{CCode}

  std::streamsize OldPrecision;
  
  OldPrecision = cout.precision(); // Genauigkeit lesen

  // Genauigkeit waehlen
  cout.precision(5);
  
  // Feldbreite waehlen
  cout.width(9);
  
  // Fuellzeichen waehlen
  cout.fill('*');


  cout << 1.23; // Ausgabe im gewaehlten Format 
                // gibt ****1.23000 aus

\end{CCode} 

width() gilt nur für die nächste Ausgabe und wird danach wieder
auf 0 zurückgesetzt (Nur soviele Zeichen wie nötig ausgeben).


Mit \OviIndex{Manipulatoren} kann das Formatieren und Steuern des
Ausgabestreams vereinfacht werden. Manipulatoren sind eine geschickte Kombination von
Operator Überladung, Templates, Hilfsfunktion und Funktionsobjekt. Für viele Anwendungen
gibt es bereits vordefinierte Manuipulatoren, es ist aber auch möglich,
selbst neue Manipulatoren zu erzeugen.

\begin{CCode}


  // Format waehlen und Ausgabe in der selben Zeile
  cout <<  setprecision(5) << setw(9) << setfill('*') << 1.23;

  //  Zahl hexadezimal ausgeben (ersetzt setf())
  cout << hex << 123545;

\end{CCode} 

Einige Manipulatoren: flush, endl, setbase(), dec, hex, oct, left, right

\section{Zeiger und Referenzen}
\OviIndex{Zeiger} und \OviIndex{Referenzen} sind fast dasselbe, sie verweisen jeweils auf ein Objekt ihres Typs. Der Unterschied besteht darin, das Zeiger im Laufe ihres Lebens auf verschiedene Objekte zeigen können, oder auch auf nichts, während Referenzen bei der Erzeugung mit einem Objekt initialisiert werden müssen und anschliessend während ihrer ganzen Lebensdauer immer auf dieses Objekt verweisen. Einem Zeiger hingegen muss vor erstmaligem Gebrauch die Adresse eines passenden Objektes zugewiesen werden. Die \OviIndex{Adresse} eines Objektes erhält man mit dem Adressoperator \&. Um auf das Objekt zuzugreifen, auf das ein Zeiger verweist, muss man den Zeiger mit * \OviIndex{dereferenzieren}.
Referenzen werden insbesondere zur Parameterübergabe bei Funktionen und Methoden benutzt. Dadurch wird
Call by Reference erreicht, also das Kopieren von Argumenten unterdrückt.

\begin{CCode}
int j, k, l;    // Drei Integer Variablen definieren
int &r = j;     // Eine Referenz auf j definieren
int *ptr;       // Ein Zeiger auf int definieren

j = 100;        // Werte an j, k und l Zuweisen
k = 200;
l = 300;

cout << r;      // Wert ausgeben der von r referenziert wird (j)

ptr = &k;       // Addresse von k an Ptr zuweisen
cout << *ptr;   // Wert, auf den Ptr zeigt, ausgeben

ptr = &l;       // Addresse von l an Ptr zuweisen
cout << *ptr;   // Wert, auf den Ptr zeigt, ausgeben

void DoSomething(int &a, const int &b) 
{
   a = b + 3; // b ist const Referenz, also nur lesbar
              // a ist auch schreibbar
}

DoSomething(j, k); // j wird von DoSomething veraendert

\end{CCode}

\section{Konstantheit (const)}
Variablen und Objekte, die nur gelesen werden dürfen, kann man als \OviIndex{const}\index{Konstantheit} definieren, danach erlaubt der Compiler keine schreibenden Zugriffe auf solche Objekte mehr.

\begin{CCode}
int a;
const int ci = 4;   // Konstante, kann nicht geaendert werden
a = ci + 3;         // OK, nur lesen
ci = 19;            // Error, aendern nicht erlaubt
\end{CCode}

Konstante Objekte können nur an Zeiger oder Referenzen auf Konstante Objekte übergeben werden, sonst wäre es ja möglich, die Konstantheit zu umgehen. Nichtkonstante Objekte dürfen jederzeit an Zeiger oder Referenzen auf konstante Objekte zugewiesen werden.
\begin{CCode}
int a;
const int ci = 4;     // Konstante, kann nicht geaendert werden
const int &cr = ci;   // Referenz auf konstantes int, zeigt auf ci, alles OK
const int *cptr;      // Zeiger auf konstantes int
int * const ccp = &a; // Konstanter Zeiger auf int (const muss nach * sein)
int *ptr;             // Zeiger auf int
int &r = ci;          // Error, ci koennte sonst via r geaendert werden
const int &cra = a;   // Referenz auf konstantes int, zeigt auf a, alles OK
cr = 19;              // Error, aendern nicht erlaubt
cra = 17;             // Error, aendern nicht erlaubt (cra ist konstante Referenz)
ptr = &ci;            // Error, ci ist konstant, *ptr nicht
cptr = &ci;           // Ok (Zeiger selbst ist nicht konstant)
*cptr = 4;            // Error, Zeiger zeigt auf Konstant
cptr++;               // Ok (Zeiger selbst ist nicht konstant)
*ccp = 4;             // Ok, Zeiger zeigt auf nicht konstant)
ccp++;                // Error Zeiger ist konstant

void f1(int &i) 
{ 
   i++; 
}

void f2(const int &i) 
{ 
   i++; // Error, i ist Referenz auf konstantes int
}

f1(ci); // Error, Argument von f1 ist nich konstant, koennte i aendern
f2(ci); // Ok f2 kann i nicht aendern
\end{CCode}

Auf konstante Objekte dürfen nur konstante Methoden angewendet werden, also Methoden, die in der Klassendeklaration als const deklariert wurden. Konstante und nichtkonstante Methoden mit demselben Namen und der selben Art und Anzahl von Argumenten gelten als unterschiedlich, mit const kann man eine Methode also überladen.
\begin{CCode}
class MyClass {

int i;

  public:
    int  Test(void);         // Nur-Lese Methode, aber nicht als const deklariert
    int  Read(void) const;   // Nur-Lese Methode, ok fuer konstante Objekte
    void Write(int n);       // Schreibe Methode, nicht fuer konstante Objekte
};

int MyClass::Test(void) 
{ 
   return (i == 0);
}

int MyClass::Read(void) const
{ 
   i = i / 2;  // Error, Methode ist const, darf Attribute nicht aendern.
   return i;   // Ok, i wird nur gelesen
}

void MyClass::Write(int n) 
{ 
   i = n;  //Ok, schreiben in nicht konstanten Methoden erlaubt
}

void f1(const MyClass& mc)    // f1 veraendert Argument nicht 
{
   int i = mc.Read();  //Ok, Read() ist Konstant, nur lesen
   int j = mc.Test();  //Error, Test() ist nicht Konstant
   mc.Write();         //Error, Write() ist nicht Konstant 
}
\end{CCode}

\section{Umgehung der Konstantheit (Mutable)}
Hin und wieder hat man das Problem, dass ein Objekt zwar konstant ist,
aber eine Methode zu Verwaltungszwecken dennoch ein Attribut
(z. B. ein statischer Zugriffszähler) modifiziert.

Für diese Fälle kann ein Attribut als \OviIndex{mutable} gekennzeichnet werden. 
Attribute mit dem Qualifizierer dürfen, im Gegensatz zu 'normalen'
Attributen, von const Methoden verändert werden.


\begin{CCode}
class MyClass {

mutable int NumberOfReadAccesses;
int i;

  public:
    int  Read(void) const;   // Nur-Lese Methode, ok fuer konstante Objekte
    void Write(int n);       // Schreibe Methode, nicht fuer konstante Objekte
};

int MyClass::Read(void) const
{ 
   NumberOfReadAccesses++;  // OK, mutable darf von const-Methode geaendert werden 
   i = i / 2;  // Error, Methode ist const, darf Attribute nicht aendern.
   return i;   // Ok, i wird nur gelesen
}

void MyClass::Write(int n) 
{ 
   i = n;  //Ok, schreiben in nicht konstanten Methoden erlaubt
}

\end{CCode}
 
Man kann so ein konstantes Objekt schaffen, bei dem doch nicht ganz
alles konstant ist. Mutable sollte nur in gut begründeten Ausnahmefällen
eingesetzt werden, normalerweise wenn das Objekt gegen aussen, vom Anwender aus gesehen 
nicht ändert, aber interne Verwaltungsinformation geändert werden
muss. 

Eine mögliche Anwendung wäre ein Proxy-Objekt, das beim Lesen
effektiv Daten von einer Datei in einen lokalen Cache lädt, oder die
Linkzeiger bei Listenelementen. Das 'Umhängen' eines Listenelementes
ändert seinen Wert schliesslich nicht.

\section{switch()}
Auswahlanweisung, der Programmfluss wird bei dem case - statement weitergeführt, das mit dem Wert der switch-Anweisung übereinstimmt. Es können nur Konstanten bei den Case Zweigen stehen, es können keine Bereiche (wie 3-7) angegeben werden. Das Programm verlässt die Switch-struktur, sobald es auf ein break oder das Ende der Struktur trifft. Wenn nach einem case Statement kein break steht, fährt das Programm mit den Anweisungen des nächsten case weiter. (Sollte vermieden werden). Der Defaultfall kann weggelassen werden.
\begin{CCode}
switch (i) {
  case 1: cout << "Eins" << endl;
      break;
  case 2: cout << "Zwei" << endl;
      break;
  case 3: cout << "Drei" << endl;
      break;
  default: cout << "Was anderes" << endl;
      break;
}
\end{CCode}

\section{Funktionen}
Bei C/\Cpp findet die Übergabe von Argumenten grundsätzlich 'by Value' statt, das heisst es werden lokale Kopien erzeugt. Wenn man innerhalb der Funktion die Argumente verändert sind davon nur die Kopien betroffen, die Variablen des aufrufenden Programms bleiben unverändert. Wenn man auf Variablen des aufrufenden Programmes zugreifen will, muss man Zeiger oder Referenzen verwenden. Funktionen die keinen Rückgabewert haben (Prozeduren) werden als void deklariert. Referenzen auf Objekte, die von der Funktion gelesen, aber nicht verändert werden, deklariert man am besten als Konstant (const).

Mit 'return x' kann ein Rückgabewert aus der Funktion zurückgegeben werden, mit 'return' oder 'return x' kann eine Funktion auch vor ihrem Ende verlassen werden.


\begin{CCode}
int Mittelwert(int a, int b) 
{
  int m;
  m = (a + b) / 2;
  return (m)
}
\end{CCode}

\begin{CCode}
void Sort(int & a, int & b) 
{
  int t;
  if (a > b) {
     t = a;  // Tauscht die Originalvariablen
     a = b;  // vom aufrufenden Programm
     b = t
  }   
}
\end{CCode}

\begin{CCode}
int Maximum(int a, int b)
{
  if (a > b) {
   return (a);
  } else {
   return (b);
  }
}
\end{CCode}

\begin{CCode}
void Print( const int &a) // Referenzen waeren bei Objekten sinnvoller als hier
{
  cout << a;
}
\end{CCode}

Lokale Variablen von Funktionen besitzen zufälligen Inhalt, sie müssen vor erstmaligem Gebrauch initialisiert werden. 

Rekursion ist in \Cpp ohne Einschränkungen möglich (Solange der Speicher ausreicht).

\subsection{Prototypen}
Bevor eine Funktion verwendet wird, muss sie bekannt sein. Entweder man definiert die Funktion vor ihrer Verwendung, oder man deklariert sie mit einem \OviIndex{Prototyp}. Prototypen von globalen Funktionen werden normalerweise im Headerfile des entsprechenden Modules deklariert, lokale Funktionen am Anfang des Modules. Ein Prototyp ist nichts anderes als ein Funktionskopf, abgeschlossen mit einem Semikolon; der Funktionsrumpf fehlt. Derselbe Prototyp kann beliebig oft deklariert werden, solange sich die verschiedenen Deklarationen nicht widersprechen.

Ein Prototyp wird wie folgt deklariert:
\begin{CCode}
void Afunction(int a, char *b, int c);  // Prototyp
int Bfunction(void);                    // Prototyp
extern void PrintIt(int a);             // Prototyp, extern zeigt an dass die 
                                        // Funktion in anderem Modul definiert wird.
                                        // Uebliche Form in Headerdateien, obwohl 
                                        // extern in diesem Fall optional ist
\end{CCode}

\subsection{Defaultargumente}
Funktionen können mit \OviIndex{Defaultargumenten} ausgestattet werden. Beim Aufruf
der Funktion können diese Argumente weggelassen werden, der Kompiler setzt dann
automatisch die angegebenen Defaultwerte ein. Die Defaultargumente müssen bei der
Funktionsdeklaration (Prototyp) angegeben werden, nicht aber bei der Definition.

Die Defaultargumente müssen zwingend von hinten nach vorne angegeben werden.
(D.h. es darf nur das hinterste, das hinterste und das zweithinterste, 
das hinterste, das zweithinterste und das dritthinterste, [und so fort] Argument
weggelassen werden.)


Defaultargumente werden wie folgt deklariert:
\begin{CCode}

void Afunction(int a, char *b = NULL, int c = 17);  // Prototyp

   AFunction(1, "w", 39);  // Normaler Aufruf 

   AFunction(1, "X");      // Fuer c wird 17 benutzt 

   AFunction(1);           // Fuer b wird NULL und fuer c wird 17 benutzt 

\end{CCode}

\subsection{Inline-Funktionen}
Kurze und oft benutzte Funktionen können als \OviIndex{inline} definiert werden (Üblicherweise
wird die ganze Funktion samt Code in diesem Fall in der Headerdatei definiert). Solche Funktionen
werden (Soweit möglich) vom Compiler an der Aufrufstelle in den Code eingefügt, und nicht
aufgerufen. Dies führt zu effizienterem und schnellerem Code, kann dafür mehr
Speicherplatz benötigen, da der Code an jeder Aufrufstelle eingefügt wird. Bei langen
Funktionen ist inline nicht sinnvoll, da der Geschwindigkeitsgewinn minimal, dafür
der Speicherverbrauch gross ist.

Wenn Inlinefunktionen über Pointer oder als virtuelle Funktionen eingesetzt werden, findet kein Inlining statt (Aber sehr wohl an Stellen bei denen die Funktion direkt aufgerufen wird) 

\begin{CCode}
// in Header-Datei

inline int Square(int i)     // Inline Methoden können auch in der 
{                            // implementationsdatei (.cpp) definiert
   return i*i;               // werden, dann aber nur innerhalb dieser
                             // Datei ab Definitionspunkt verwendet werden.
}
\end{CCode}

\chapter{Klassen}
\section{Was sind Klassen}
Eine \OviIndex{Klasse} unter \Cpp  entspricht im wesentlichen einer Struktur in C oder einem Record in Pascal. Nur dass auch die Funktionen (Methoden), die mit den Daten (Attribute) der Klasse arbeiten zu der Klasse hinzugefügt werden und nicht davon separiert sind.

Also wird aus folgendem Code unter C:
\begin{CCode}
// h-File
struct Datum {
   int Tag;
   int Monat; 
   int Jahr;
};

void InitDate(Datum *Date, int NewTag, int NewMonat, int NewJahr);
void PrintDate(Datum *Date);
void AddDayToDate(Datum *Date, int Day);
\end{CCode}

\begin{CCode}
//c-Datei
void InitDate(Datum *Date, int NewTag, int NewMonat, int NewJahr)
{
   Datum->Tag   = NewTag;
   Datum->Monat = NewMonat;
   Datum->Jahr  = NewJahr;
}

int main(int argc, char *argv[]) {

   Datum DasDatum;                    // Erzeugen des Objektes
   InitDate(&DasDatum, 24, 12, 2002); // Initialisieren des Objektes
   PrintDate(&DasDatum);              // Ausgeben des Objektes
}
\end{CCode}


folgender Code in \Cpp:
\begin{CCode}
// h-File
class Datum {
   int Tag;
   int Monat; 
   int Jahr;

   void Init(int NewTag, int NewMonat, int NewJahr);
   void Print();
   void AddDay(int Day);
};
\end{CCode}

\begin{CCode}
//cpp-Datei
void Datum::Init(int NewTag, int NewMonat, int NewJahr)  // Init von Datum
{
   Tag   = NewTag;
   Monat = NewMonat;
   Jahr  = NewJahr;
}

int main(int argc, char *argv[]) {

   Datum DasDatum;                  // Erzeugen des Objektes
   DasDatum.Init(24, 12, 2002);     // Initialisieren des Objektes
   DasDatum.Print();                // Ausgeben des Objektes
}
\end{CCode}


\section{Konstruktoren / Destruktoren}
Unter \Cpp ist es nicht nötig, nach dem Erzeugen eines Objektes noch explizit eine Initialisierungsfunktion aufzurufen. Dafür sind die \OviIndex{Konstruktoren} vorgesehen, welche vom Compiler automatisch beim Erzeugen eines Objektes aufgerufen werden.
Vorheriges Beispiel sieht unter Einsatz von Konstruktoren wie folgt aus:

\begin{CCode}
// h-File
class Datum {
   int Tag;
   int Monat; 
   int Jahr;

        Datum(int NewTag, int NewMonat, int NewJahr); // Konstruktor
   void Print();
   void AddDay(int Day);
};
\end{CCode}

\begin{CCode}
//cpp-Datei
Datum::Datum(int NewTag, int NewMonat, int NewJahr)
{
   Tag   = NewTag;
   Monat = NewMonat;
   Jahr  = NewJahr;
}

int main(int argc, char *argv[]) {

   Datum DasDatum(24, 12, 2002);    // Erzeugen und Initialisieren des Objektes
   DasDatum.Print();                // Ausgeben des Objektes
}
\end{CCode}

Unter \Cpp ist es somit praktisch nicht mehr möglich, uninitialisierte Objekte im System zu haben, da jedes Objekt beim Erzeugen über einen Konstruktor initialisiert wird.

Als Gegenstück zum Konstruktor gibt es noch den \OviIndex{Destruktor}, welcher automatisch beim Löschen eines Objektes aufgerufen wird.  Die Aufgabe des Destruktor ist das Aufräumen bevor das Objekt definitiv verschwindet, also das Freigeben von eventuell reservierten Resourcen und interne Buchhaltungen aktualisieren, falls vorhanden. In vielen Fällen hat der Destruktor nichts zu tun, dann kann er einfach weggelassen werden


\subsection{Konstruktoren}
\OviIndex{Konstruktoren} sind Methoden, die vom Compiler automatisch aufgerufen werden, wann immer ein neues Objekt erzeugt wird. Die Aufgabe dieser Methoden ist, dafür zu sorgen dass das Objekt korrekt initialisiert wird. Den Konstruktor kann man daran erkennen, dass er denselben Namen wie die Klasse aufweist. Wenn vom Programmierer kein Konstruktor deklariert wird, erzeugt der Compiler selbstständig den sogenannten \OviIndex{Defaultkonstruktor}, der einfach alle Attribute auf 0 setzt. Es kann für eine Klasse mehrere Konstruktoren geben, sie müssen sich einfach in der Anzahl oder dem Typ der Argumente unterscheiden (Überladen).
\begin{CCode}
class MyClass {
  int k, j;
  public:
                MyClass(void);         // Argumentloser Konstruktor
                MyClass(int new_k);    // Ueberladen fuer int
                MyClass(float new_k);  // Ueberladen fuer float
   ...
};
\end{CCode}

Die \OviIndex{Initialisierung} der Attribute erfolgt beim Konstruktor normalerweise über die \OviIndex{Initialisiererliste}. Die Initialisiererliste wird durch ein ':' eingeleitet, die Attribute werden mit der Konstruktorsyntax initialisiert (Attributname(Wert1, Wert2, ....) ). Basisklassen und eingebettete Klassen ohne Defaultkonstruktor sowie Referenzen können \emph{nur} über die Initialisiererliste initialisiert werden.

\begin{CCode}
   MyClass::MyClass(void) : k(13), j(2)
   {
     ....    // Weiterer Code
   }

   MyClass::MyClass(int new_k) : k(new_k), j(2)
   {
     ....    // Weiterer Code
   }
\end{CCode}


Der Compiler kann zum Teil selbst nach Bedarf Konstruktoren aufrufen, um Typen
ineinander umzuwandeln. Dies kann mit dem Schlüsselwort \OviIndex{explizit} unterdrückt werden.
Mit explizit definierte Konstruktoren müssen explizit aufgerufen werden. 
\begin{CCode}

class YourClass {
  int k, j;
  public:
                YourClass(void);         // Argumentloser Konstruktor
                YourClass(int new_k);    //  Ueberladen fuer int
   ...
};


void f1(YourClass y) 
{
   ...
}

   YourClass y1 = 10;  // Ok, ruft implizit Konstruktor YourClass(int) auf

   y1 = 22;            // Ok, ruft implizit Konstruktor YourClass(int) auf

   f1(22)              // Ok, ruft implizit Konstruktor YourClass(int) auf


class MyClass {
  int k, j;
  public:
                MyClass(void);                  // Argumentloser Konstruktor
                explicit MyClass(int new_k);    // explizit Ueberladen fuer int
   ...
};


void f2(MyClass m) 
{
   ...
}

   :
   :
   MyClass m1 = 10;          // Geht nicht, impliziter Konstruktoraufruf verboten
   MyClass m2 = MyClass(10); // Ok, expliziter Konstruktoraufruf

   m2 = 22;                  // Geht nicht, impliziter Konstruktoraufruf verboten
   m2 = MyClass(22);         // Ok, expliziter Konstruktoraufruf 

   f2(22)                    // Geht nicht, impliziter Konstruktoraufruf verboten
   f2(MyClass(22));          // Ok, expliziter Konstruktoraufruf
   :
   :


\end{CCode}  

\subsection{Copy-Construktor}
Es gibt einen speziellen Konstruktor, genannt \OviIndex{Copy-Construktor}, der vom Compiler jedes mal aufgerufen wird wenn der Compiler ein Objekt kopieren muss (Argumentübergabe, Temporäre Objekte, nicht aber bei einer Zuweisung). Wenn der Programmierer keinen \OviIndex{Kopier-Konstruktor} zur Verfügung stellt, erzeugt der Compiler auch hier selbst eine entsprechende Methode.
\begin{CCode}
class MyClass {
  int k, j;
  public:
   ...
                MyClass(const MyClass &Source);  // Copy-Konstruktor
   ...
};
\end{CCode}

\subsection{C11 Move-Construktor}
Unter C11 gibt es neu den Movekonstruktor (Siehe Kapitel \autoref{subsec:RValueRefs}), welcher unnötiges kopieren vermeiden hilft und so einen effizienteren Umgang mit Temporarys erlaubt.

\subsection{Zuweisungsoperator}
In die gleiche Kategorie gehört auch der \OviIndex{Zuweisungsoperator} (operator =()), dieser wird jedesmal aufgerufen, wenn ein Objekt einem anderen zugewiesen wird. Auch hier erzeugt der Compiler selbständig eine Zuweisungsmethode, wenn dies vom Programmierer unterlassen wird. Der Zuweisungsoperator muss den Spezialfall einer Zuweisung an sich selbst behandeln, und sollte eine Referenz auf sich selbst (*this) zurückliefern. Der Zuweisungsoperator kann
für verschiedene Datentypen überladen werden. Er ist eine Kombination von Copy-Konstruktor und Destruktor, da der alte Inhalt des Objektes geloescht (Ueberschrieben) und durch neue Werte ersetzt wird.

\begin{CCode}
class MyClass {
  int k, j;
  public:
   ...
                MyClass &operator =(const MyClass &Source);  // Zuweisungs-Operator
   ...
   ...
};

MyClass &MyClass::operator =(const MyClass &Source)
{
   if (this != &Source) {
      k = Source.k;
   }
   return *this;
}

\end{CCode}

\subsection{Destruktor}
Der \OviIndex{Destruktor} ist das genaue Gegenteil des Konstruktors, er wird immer dann aufgerufen, wenn ein Objekt aufhört zu existieren. Seine Aufgabe ist es, vor dem Ableben eines Objektes noch \OviIndex{aufzuräumen}. Wenn zum Beispiel im Konstruktor eine Datei geöffnet wurde, ist es Pflicht des Destruktors, diese Datei wieder zu schliessen. Den Destruktor erkennt man daran, dass er denselben Namen wie die Klasse aufweist, allerdings mit einer vorangestellten Tilde ('\verb$~$'). Es gibt für jede Klasse nur einen Destruktor. (Destruktoren von Basisklassen werden automatisch nach Abarbeitung des Destruktors der abgeleiteten Klasse aufgerufen). Wenn kein Destruktor benötigt wird, kann er weggelassen werden.
\begin{CCode}
class MyClass {
  int k, j;
  public:
   ...
               ~MyClass(void);         // Destruktor
   ...
   ...
};
\end{CCode}

\subsection{Umwandlungsoperator}
\OviIndex{Umwandlungsoperatoren} gehören eigentlich ins Kapitel Operatoren überladen, werden aber hier erwähnt. Wenn ein Objekt in ein anderes umgewandelt werden soll, ist dies entweder mit Konstruktoren oder Umwandlungsoperatoren möglich.
Wenn man eine automatische Umwandlung einer Klasse A in einen int haben möchte, wenn ein Objekt dieser Klasse einem int zugewiesen wird, müsste folgender Umwandlungsoperator definiert werden:
\begin{CCode}
class A {
   private:
   int i;
   int j;
   
   public:
   A(int _i, int _j) : i(_i), j(_j) {};
   operator int ();  //Beachten: Keinen Rueckgabetyp angeben!!!
};

A::operator int() //Beachten: Keinen Rueckgabetyp angeben!!!
{
   return i+j;
}

int main(int argc, char *argv[]) 
{
   A x(1, 2);
   int q;
   q = x;
   cout << q;
}
\end{CCode}

Achtung: Umwandlungsoperatoren sollten sehr zurückhaltend eingesetzt werden, da sonst viele inkompatible Zuweisungen plötzlich erlaubt werden, und viele Flüchtigkeitsfehler vom Compiler nicht mehr erkannt werden können.
Es kann auch passieren, dass unerwartete Umwandlungen geschehen und man den Überblick über die möglichen Umwandlungen verliert. 


\section{Zugriffsrechte}
In Klassen kann unter \Cpp der Zugriff auf Member (Attribute und Methoden) eingeschränkt werden. Es gibt folgende \OviIndex{Zugriffsrechte}\index{public}\index{protected}\index{private}:
\paragraph{public:} 
Auf dieses Element darf von überall her zugegriffen werden, sollte für Attriubute vermieden werden (Default bei Strukturen).
\paragraph{protected:}	Auf dieses Element kann nur von Methoden dieser Klasse und davon abgeleiteten Klassen (sowie von Friendfunktionen) zugegriffen werden.
\paragraph{private:}	Auf dieses Element kann nur von Methoden dieser Klasse (sowie von Friendfunktionen) zugegriffen werden (Default bei Klassen).

Eine Zugriffsberechtigung ist solange gültig bis eine andere eingestellt wird.

\section{Friendfunktionen}
\index{Friendfunktionen}Wenn man einer Funktion oder einer Klasse Zugriff auf private oder protected Member erlauben will, ohne diese Member als public zu definieren, kann man die Funktion oder Klasse als \OviIndex{friend} deklarieren:

\begin{CCode}
class MyClass {

friend void AccessMe(MyClass *mc);  // Globale Funktion AccesMe darf auf private
                                    // Member zugreifen 
friend void OtherClass::Intrude(MyClass *mc);  // Methode intrude der Klasse 
                                               // OtherClass darf auf auf private 
                                               // Member zugreifen 
friend class BigBrother;            // Alle Methoden der Klasse BigBrother
                                    // duerfen auf private Member zugreifen 
  int i;
  int k;

  public:
    MyClass();
    MyClass(int new_i, int new_k);
   ~MyClass();
  
  int Getk();
};
\end{CCode}

Achtung, die Klasse gibt anderen das Recht auf sie zuzugreifen, und nicht umgekehrt. Die friend Deklaration muss deshalb immer in der betroffenen Klasse stehen. Mit friend kann man den Zugriffsschutz für eine bestimmte Funktion oder Klasse ausschalten. Eine Klasse kann beliebig viele Friends deklarieren.
Man sollte damit aber zurückhaltend umgehen, da damit das Prinzip der Kapselung verletzt wird.
In vielen Fällen ist es jedoch sinnvoller, ein bestimmtes Element als Friend zu deklarieren
als eine Menge von Set- und Get-Methoden einzubauen, da mit unnötigen Set- und Get-Methoden
die Kapselung erst recht durchbrochen wird.

\section{Vorwärts Deklarationen} 
\index{Vorwärts Deklarationen}Wenn man Zeiger oder Referenzen auf Objekte definieren/deklarieren will, muss die Klasse nicht vollständig bekannt sein. Solange man nicht auf Attribute oder Methoden der Klasse zugreift, reicht eine einfache Klassendeklaration der Form
\begin{CCode}
class SomeClass;

void f1(SomeClass &r);
\end{CCode}

Sobald man auf Member zugreift, muss die Klasse vollständig bekannt sein.

\section{Inline Methoden}
\index{Inline Methoden} 
Kurze und oft benutzte Methoden können als \OviIndex{inline} definiert werden. (Üblicherweise
wird die ganze Funktion samt Code in diesem Fall in der Headerdatei innerhalb der Klassendeklaration definiert). Solche Methoden
werden (Soweit möglich) vom Compiler an der Aufrufstelle in den Code eingefügt, und nicht
aufgerufen. Dies führt zu effizienterem und schnellerem Code, kann dafür mehr
Speicherplatz benötigen, da der Code an jeder Aufrufstelle eingefügt wird. Bei langen
Methoden ist inline nicht sinnvoll, da der Geschwindigkeitsgewinn minimal, dafür
der Speicherverbrauch gross ist.

\begin{CCode}
class A {
   
   private:
   int i;
   
   public:
   A(int _i) : i(_i) {};      //Konstruktor, macht nichts (Leerer Funktionsrumpf)
  ~A() {};                    //Destruktor, macht nichts (Leerer Funktionsrumpf)
   
   int GetI() { return i; };  // Inline Code
   void SetI(int ni) { if (i > 0) {i = ni;};};  // Inline Code
   void SetX(int ni) { 
                       if (i > 0) {   // Inline Code darf auch
                          i = ni;     // strukturiert geschrieben
                       };             // werden...
                     };  
   void DoSomething();                  
};


inline void A::DoSomething()  // Inline Methoden können auch
{                             // ausserhalb der Klasse definiert
   if (i > 2) {               // werden. (Sollten aber innerhalb 
      i = i * 2;              // der selben Datei definiert werden)
   }
}
\end{CCode}


 
\section{Statische Member}
\OviIndex{Statische Member}\index{Klassenmember} (Attribute oder Methoden) einer Klasse sind für alle Objekte gemeinsam, sie beziehen sich nicht auf ein bestimmtes Objekt. Statische Methoden werden auf die Klasse angewendet, nicht auf ein Objekt. Auf statische Member wird über die Klasse mit dem Scope-Resolutions-Operator ('::') zugegriffen.

\begin{CCode}
class A {
   private:
   int i;
   static int Count;   // Zaehlt anzahl Objekte
   
   public:
   A(int _i) : i(_i) { Count++; };      //Konstruktor, zaehlt Objekte
   A(const A &r) : i(r.i) { Count++; }; //Copy-Konstruktor, zaehlt Objekte
  ~A() { Count--; };                    //Destruktor, zaehlt Objekte
   
   static int GetNumberOfObjects() { return Count; };
};

int A::Count = 0;   // Definiere und initialisiere statische Member-Variable 
                    // Nur an einem Platz im ganzen Programm, normalerweise
                    // im .cpp-Modul der Klasse

A a1(1), a2(2), a3(3);   // Einige Objekte erzeugen
int Number = A::GetNumberOfObjects(); // Anzahl Objekte abfragen
\end{CCode}


\section{Der this-Pointer}
Jede Methode (Auch Kostruktoren, Destruktor und Operatorfunktionen) haben implizit einen vordefinierten Pointer mit dem Namen \OviIndex{this}, dieser zeigt immer auf das Objekt, auf welches die Methode angewendet wurde. Damit hat jede Methode Zugriff auf die Adresse des aktuellen Objektes, und kann im Notfall auch über diesen Zeiger auf die Attribute zugreifen.

\begin{CCode}
class Auto {
   
     int Gewicht;
   
   public:
     void SetGewicht(int Gewicht);
}

void Auto::SetGewicht(int Gewicht)
{
   Gewicht = Gewicht;       //  Aehm, welches Gewicht jetzt????  (beide mal 
                            //  der selbe Name!)
   this->Gewicht = Gewicht; //  Ah nun ist alles klar. Es sollte aber 
                            //  dennoch vermieden werden Attributnamen bei 
                            //  Funktionsparametern zu verwenden, da der
                            //  Parameter das Attribut verdeckt
}
\end{CCode}
 
\section{Zeiger auf Elemente (Memberpointer)}
In \Cpp gibt es die Möglichkeit, \OviIndex{Memberpointer} zu verwenden. Das sind Zeiger
auf die einzelnen Elemente einer Klasse. Ein Memberpointer enthält in dem Sinne
eigentlich nicht eine Adresse, sondern mehr ein Offset oder die Identifikation 
eines Klassenmembers. Ein \OviIndex{Elementzeiger} kann nur für die Klasse verwendet werden, für
die er definiert wurde. Ein Memberzeiger ist also an eine bestimmte Klasse gebunden.
\begin{CCode}
class Demo {
   public:
   
   int  a;
   int  b;
   int  c;
   char z;
   
   void f1();
   void f2();
   void f3();
};

Demo   x;
Demo   y;
Demo *dp = &y;

int    Demo::*MemberPointer;

MemberPointer = &Demo::a;       // Zeigt nun auf a der Klasse Demo 
  
x.*MemberPointer   = 5;         // Weist x.a 5 zu;
dp->*MemberPointer = 5;         // Weist y.a 5 zu;

MemberPointer = &Demo::b;       // Zeigt nun auf b der Klasse Demo 

x.*MemberPointer   = 6;         // Weist x.b 6 zu;
dp->*MemberPointer = 6;         // Weist y.b 6 zu;

MemberPointer = &Demo::z;       // Fehler, z ist nicht int

void (Demo::*MemberfPointer)(); // Zeiger auf Memberfunktion

MemberfPointer = &Demo::f1;     // Zeigt nun auf f1() der Klasse Demo 

(x.*MemberfPointer)();          // Ruft die Methode f1() fuer x auf;
(dp->*MemberfPointer)();        // Ruft die Methode f1() fuer y auf;

MemberfPointer = &Demo::f2;     // Zeigt nun auf f2() der Klasse Demo 

(x.*MemberfPointer)();          // Ruft die Methode f2() fuer x auf;
(dp->*MemberfPointer)();        // Ruft die Methode f2() fuer y auf;
\end{CCode}
Wie schon bei den 'normalen' Funktionszeigern wird dringend der Gebrauch von
typedef empfohlen, um den Code lesbarer zu halten. 

\section{Erzeugen von Objekten}
Unter \Cpp werden Objekte wie gewöhnliche Variablen erzeugt. Ein Objekt ist somit nichts anders als eine Variable vom Typ seiner Klasse.

Eine int Variable kann wie folgt erzeugt werden:

\begin{CCode}
  int i;       // Erzeugen einer Integervariablen mit dem Namen i
  int *p;      // Erzeugen eines Zeigers auf einen Integer
  p = new int; // Dynamisches Erzeugen einer namenlosen Integervariablen und
                  den Zeiger p darauf zeigen lassen
\end{CCode}

Ein Objekt der Klasse Student wird entsprechend wie folgt erzeugt:

\begin{CCode}
  Student s;       // Erzeugen eines Studentenobjektes mit dem Namen s
  Student *sp;     // Erzeugen eines Zeigers auf einen Studenten
  sp = new Student; // Dynamisches Erzeugen eines namenlosen Studenten-
                    //  objektes und den Zeiger sp darauf zeigen lassen
                      
  Student s2("Hans", 27);         // Mit Argumenten fuer Konstruktor                    
  sp = new Student("Fritz", 28);                      
\end{CCode}

Wenn Variablen und Objekte normal erzeugt werden (Also nicht dynamisch mit new), dann werden sie automatisch gelöscht wenn sie nicht mehr gültig sind, also der Programmlauf ihren Gültigkeitsbereich verlässt. (Lokale Variablen sind innerhalb des Blockes gültig, in dem sie definiert (erzeugt) wurden).
Globale Objekte werden vor Aufruf von main() erzeugt, auch deren Konstruktor wird vor dem Start von main() aufgerufen. Destruktoraufruf und Zerstörung globaler Objekte erfolgt nach Beendigung von main() resp des Programms. 

Werden Variablen oder Objekte \OviIndex{dynamisch erzeugt} (Was bei Objekten sehr oft der Fall ist), müssen sie vom Programmierer selbst bei Nichtgebrauch mit delete wieder gelöscht werden (Im Gegensatz zu Java). Werden die Objekte nicht gelöscht, verbleiben sie im Speicher und belegen so Resourcen.

Objekte können selbst Objekte enthalten, hier gilt dasselbe wie oben. Wenn die Objekte wie gewöhnliche Membervariablen definiert sind, werden diese ebenfalls automatisch erzeugt wenn das übergeordnete Objekt erzeugt wird.
Beispiel:

\begin{CCode}
class StudentClass {
   int Alter;
   AdressClass Adresse; // Eingebettetes Objekt, ist Attribut 
                        // der Klasse StudentClass
};
\end{CCode}

Soll das eingebettete Objekt selbst aber dynamisch erzeugt werden, muss dies durch den Konstruktor erledigt werden, und das Löschen des Unterobjektes muss im Destruktor der übergeordneten Klasse erfolgen. Dies unabhängig davon ob die übergeordnete Klasse selbst dynamisch oder statisch erzeugt wurde.
Beispiel:
\begin{CCode}
class StudentClass {
   int Alter;
   AdressClass *Adresse; // Eingebettetes Objekt, wird via Zeiger referenziert und

\section{Placement new}

// Weiterer Code

// Das Objekt wird nicht mehr gebraucht, manueller Destruktoraufruf
sp->~StudentClass();

\subsection{Vererbung}
Um eine existierende Klasse zu erweitern, kann eine neue Klasse erstellt werden, die von der existierenden Klasse erbt. 

Bei der \OviIndex{Vererbung} übernimmt die neue Klasse zuerst einmal alle Attribute und Methoden der bereits bestehenden Klasse, und fügt eventuell neue Attribute und Methoden hinzu, oder ändert die geerbten Methoden ab (Überschreibt Sie).

Wenn eine Klasse von einer bestehenden Klasse (Basisklasse) erben soll, wird nach dem Klassennahmen ein Doppelpunk gesetzt, anschliessend folgt ein Zugriffspefizierer und danach der Namen der Basisklasse. Der Zugriffsspezifizierer legt die Zugriffsrechte für die Basisklasse fest, genau so wie wenn sie ein Attribut der neuen Klasse wäre. Normalerweise wird hier public verwendet, da sonst Probleme mit \OviIndex{Polymorphismus} auftreten. (Die Methoden der Basisklasse sind via abgeleitete Klasse nicht zugänglich)

Unter \Cpp sieht eine \OviIndex{Vererbung} wie folgt aus:

\paragraph{ExtendedClass.hpp} (Klassendeklaration)

\subsection{Polymorphismus}
Um unter \Cpp \OviIndex{Polymorphismus} zu ermöglichen, müssen die betroffenen Methoden der Basisklasse als \OviIndex{virtuell} deklariert werden. Die Deklaration als virtuell erfolgt in der Klassendeklaration durch das Schlüsselwort \OviIndex{virtual}, aber nicht bei der Definition (Implementation).
\emph{Sobald bei einer Klasse eine Methode virtuell ist, sollte auch der Destruktor als virtuell deklariert werden. Sogar dann, wenn in der Basisklasse eigentlich kein Destruktor nötig wäre!} Wenn man dies unterlässt, handelt man sich meist Probleme ein, weil unter Umständen sonst der falsche Destruktor aufgerufen wird. (Spätestens in der Wartungsphase bei Erweiterungen). Es soll ja schliesslich der Destruktor des aktuellen Objektes und nicht der der Basisklasse benutzt werden.

Wenn es keinen Sinn macht, von der Basisklasse ein Objekt anlegen zu können, weil es keine Entsprechung in der realen Welt hat (Weil es z. B. eine Generalisierung ist), kann man sie als \OviIndex{abstrakte Klasse} deklarieren. Dazu muss man unter \Cpp mindestens eine Methode als \OviIndex{rein virtuell} deklarieren (mit = 0 in der Deklaration: 'virtual void f(void) = 0'). Danach muss die Funktion für die Basisklasse auch nicht implementiert werden. Es ist ja jetzt schliesslich nicht mehr möglich, Objekte der Basisklasse zu erzeugen.

\section{Überladen von Funktionen}
Unter \Cpp können Funktionen überladen \index{Überladen von Funktionen} werden. Das heisst, es kann mehrere Funktionen mit demselben Namen geben, solange sie sich durch die Anzahl oder der Art der Argumente unterscheiden.
\begin{CCode}
int max(int a, int b)
{
   if (a > b) {
      return a;
   } else {
      return b;
   }
}

float max(float a, float b)
{
   if (a > b) {
      return a;
   } else {
      return b;
   }
}

int main (int argc, char *argv[])
{
   int x, y;
   float q, r;
   q = max(x, y) // Ruft max() fuer int auf (Rueckgabewert hat keinen Einfluss 
                 // auf Auswahl)
   x = max(q, r) // Ruft max() fuer float auf (Rueckgabewert hat keinen Einfluss 
                 // auf Auswahl)
}
\end{CCode}

 
\section{Überladen von Operatoren}
\index{Überladen von Operatoren}\OviIndex{Operatoren} ( + - * / etc.) gelten in \Cpp als normale Funktionen mit dem Namen operator x, wobei für x jeweils der gewünschte Operator eingesetzt wird. Im speziellen können diese Operatoren für eigene Typen (Also Klassen, resp. Objekte überladen werden)

\begin{CCode}
class Complex {
   float r, i;
   public:
   Complex (int n_r, int n_i) : r(n_r), i(n_i) {};
   Complex & Operator += (Complex &b) {r += b.r; i += b.i; return *this; };
   Complex & Operator -= (Complex &b) {r -= b.r; i -= b.i; return *this; };
   friend Complex operator +(Complex &a, Complex &b)
   friend Complex operator -(Complex &a, Complex &b)
   // To Do: A lot of other operands
};

inline Complex operator +(Complex &a, Complex &b) 
{
   return Complex(a.r + b.r, a.i + b.i);
}

inline Complex operator -(Complex &a, Complex &b) 
{
   return Complex(a.r - b.r, a.i - b.i);
}


int main(int argc, char *argv[]) 
{
   // Create some Complex Objects
   Complex x(1, 0);
   Complex y(0, 1);
   Complex z(0, 0);
   Complex q(0, 0);

   // Do some Complex Operations 
   z = x + y;   // is same as z = operator +(x, y)
   q = z - y;   // is same as q = operator -(x, y)
   z += x;      // is same as z.operator +=(x)
   q = z - x + y;  // is same as q = operator -(z, operator +(x, y))
 
   return 0;
}
\end{CCode}

\OviIndex{Operatorfunktionen} können als Klassenmember oder als globale (friend-) Funktionen definiert werden, man nimmt normalerweise die Form, die besser zur Aufgabe passt. Wenn man die Klassenmemberfunktion benutzt, muss man beachten, dass der erste Operand immer ein Objekt sein muss.
Der Ausdruck a + b enspricht somit den Ausdrücken a.operator +(b) oder operator +(a, b), man sollte aber nur eine von beiden Varianten pro Klasse und Operator benutzen.

Mann muss beachten, dass alle Operatoren ausser den Zuweisungsoperatoren ein neues Objekt als Ergebnis der Operation zurückliefern müssen, da die Operanden ja nicht verändert werden sollten. Bei Zuweisungsoperatoren sowie \OviIndex{Inkrement} und \OviIndex{Dekrement} wird der erste Operand verändert, das ist ja schliesslich der Sinn dieser Operatoren. Hier wird normalerweise nur eine Referenz auf den ersten Operand zurückgeliefert (Ausnahme Post-Inkrement/Dekrement, da hier eine Kopie des unveränderten Objektes geliefert werden muss).
Bei den Inkrement- und Dekrementoperatoren gibt es eine weitere Spezialitaet. Um Post (i++) und Pre (++i)
auseinenderzuhalten wird ein zusätzliches (bedeutungsloses) Argument eingeführt:

{
	\centering
		\begin{tabular}{ll}
      prefix (++i): &operator++()\\
      postfix (i++):&operator++(int)\\
      prefix (--i): &operator--()\\
      postfix (i--):&operator--(int)\\
		\end{tabular}
   
   Das int Argument hat keine Bedeutung, es dient nur zur Unterscheidung der Post- und Prefixversionen
}   
 
Allgemein gilt: wann immer möglich eine Referenz statt ein Objekt übergeben, dies führt zu effizienterem Code. (Noch besser ist, wenn wann immer möglich konstante Referenzen benutzt werden)
 
Die Operatoren new und delete lassen sich ebenfalls überladen, dadurch lässt sich eine eigene Speicherverwaltung implementieren, dabei kann das globale new/delete, oder der operator new/delete für eine bestimmte Klasse überladen werden:

\begin{CCode}
class SpecialMemory {
   float r, i;
   public:
  
   // A lot of methods

   // Our overloaded new and delete
    void *operator new (size_t size) {
       // some Magic to get memory
       // in a better way than mewe usually does
    
       return AdressOfNewMemory;
    };
    
    void *operator new[] (size_t size) {
       // some Magic to get memory
       // in a better way than mewe usually does
    
       return AdressOfNewMemory;
    };
    
    void  operator delete(void *pMem, size_t size) {

    };

Es ist auch möglich, das Placement-new zu überladen, oder neue Versonen von new mit anderen Argumentenlisten zu definieren.



 

\section{Ausnahmebehandlung (Exceptions)}
Oft sind der Ort, an dem ein Fehler entsteht, und der Ort, wo er angemessen behandelt werden kann, an verschiedenen Stellen im Programm zu finden. Dies lässt das behandeln von Fehlern meist zu einem komplizierten und fehleranfälligen Problem werden.

Um die Fehlerbehandlung zu vereinfachen, wurde das \OviIndex{Exceptionhandling} in \Cpp eingeführt. Wenn irgendwo im Programmlauf ein Problem auftritt, welches an dieser Stelle nicht behandelt werden kann, wirft man mit throw() eine \OviIndex{Ausnahme}. Damit wird die normale Abarbeitung des Programmes abgebrochen und zum nächsten passenden Fehlerbehandlungsteil gesprungen. Dieser kann innerhalb der selben Funktion sein, aber auch in einer der aufrufenden Funktionen. Dabei wird die Abarbeitung der aktuellen Funktion(-en) abgebrochen und in der Aufrufhierarchie (=Verschachtelung von Aufrufen) zurückgekehrt, bis ein passender \OviIndex{Handler} gefunden wird. Falls kein passender Handler existiert, wird das Programm mit einer Laufzeitfehlermeldung abgebrochen.

Mit \OviIndex{throw} werden normalerweise Objekte geworfen, es kann aber im Prinzip jeder Datentyp geworfen werden (Also auch int, char, ...). Beim Fangen wird jeweils der Handler aktiv, der den entsprechenden Datentyp auffangen kann.
Handler für Basisklassen fangen auch Objekte von davon abgeleiteten Klassen. Der Handler catch(...) fängt jede beliebige Exception. Nach jedem \OviIndex{try}-Block folgt eine Liste von Handlern (catch()). Die Handler werden in der Reihenfolge ihrer Definition durchsucht,  der erste passende Handler wird benutzt, und alle weiteren Handler
dieser Liste ignoriert. Somit werden alle Handler nach \OviIndex{catch}(...) ignoriert. 

Funktionen (und Methoden) können in der Deklaration angeben, ob und welche Exceptions sie werfen. Dazu wird das Schlüsselwort throw() verwendet. Im throw Statement werden alle Exceptions aufgeführt, welche
von der Funktion ausgelöst werden können. Ein fehlendes throw() Statement bedeutet, dass diese
Funktion ohne Einschränkung alle Exceptions werfen kann, ein leeres throw() bedeutet, dass die
Funktion keine Exception werfen kann. Der Compiler versucht bereits zur Laufzeit alle erkennbaren
Widersprüche zu reklamieren. Wenn eine Funktion zur Laufzeit versucht, eine nicht deklarierte
Exception zu werfen, wird das Programm mit einer Laufzeitfehlermeldung abgebrochen.

Beispiel:
\begin{CCode}
class ErrorClass {
};

class NegativeSquareRootErrorClass : public ErrorClass{
};

class DivideByZerroErrorClass : public ErrorClass{
};

int ThrowAnything(int x) // Can throw any exception
{
  ...
}

int ThrowNothing(int x) trow() // Can throw no exception, must catch all
{                              // internal generated exception
  ...
}


int SquareRoot(int x) throw(NegativeSquareRootErrorClass)  {
   if (x < 0) {
      throw (NegativeSquareRootErrorClass());
   }
   int result = x / 2;
   // some clever square root algorithm
   return (result);
}

int Inverse(int x) throw(DivideByZerroErrorClass) {
   if (x == 0) {
      throw(DivideByZerroErrorClass());
   }
   return (1/x);
}

int f(int x1, int x2) throw(NegativeSquareRootErrorClass){
   int Result;
   try {
      int r1 = Inverse(x1);
      int r2 = Inverse(x2);
      Result = SquareRoot(r1) + SquareRoot(r2);
      Result = Inverse(Result);
   }
   catch (DivideByZerroErrorClass &e) {
      Result = 0;
   }
   return Result;
}

int main (int argc, char *argv[]) {
   int a, b;
   cin >> a;
   cin >> b;
   
   try {
      cout << f(a,b);
   }
   catch (DivideByZerroErrorClass &e) {
      cout << "Division durch Null";
   }
   catch (NegativeSquareRootErrorClass &e) {
      cout << "Wurzel aus negativer Zahl";
   }
   catch (...) {
      cout << "Unerwartete Exception";
   }
}
\end{CCode}


Sämtliche Objekte, deren Gültigkeitsbereich während der Suche nach dem passenden Handler verlassen wird, werden vom Compiler automatisch gelöscht, genau so wie wenn der Bereich auf normalem Wege verlassen worden wäre. Dadurch ist gewährleistet, dass für automtisch (statisch) erzeugte Objekte keine Resourcenlecks entstehen. Für dynamisch erzeugte Objekte (new) ist der Benutzer hingegen selbst zuständig (Durch entsprechende catch throw Anweisungen).

Falls in einer Funktion Resourcen alloziert wurden (Z.B. mit new), muss der Programmierer selbst für deren Freigabe im Fehlerfall sorgen. Er muss unter Umständen alle Exceptions auffangen, und in der Handlerroutine die in dieser Funktion allozierten Ressourcen wieder freigeben. Wenn die Exception in diesem Handler nicht behandelt werden kann, wird sie mit throw() einfach erneut geworfen:

\begin{CCode}
void f(int y) {

   int *ptr;
   // Speicher reservieren
   ptr = new int [100];
   try {
      // Aufruf einer Funktion, die eine Exception werfen könnte
      PossibleThrowingFunction(y, ptr);
   }
   catch (...) {
      // jede Exception auffangen
      // Speicher freigeben
      delete[] ptr;
      // und exception erneut werfen, wir wollen sie nicht hier behandeln,
      // wir haben sie nur gefangen, um den Speicher im Fehlerfall auch
      // freizugeben
      throw;
   }
   
   // Aufruf einer Funktion, die keine Exception werfen kann
   NoThrowing(ptr);
   
   // Speicher wieder freigeben
   delete[] ptr;
}
\end{CCode}

\section{Laufzeit Typinformation (RTTI run-time type information)}
\index{RTTI}
\index{run-time type information}
\index{Laufzeit Typinformation}
Mit dem Operator \OviIndex{typeid}() ist es möglich, zur Laufzeit Informationen
über den Typ eines Objektes zu erhalten. Der Operator typeid liefert eine Referenz auf ein Objekt der Klasse \OviIndex{type\_info}, welche in \mbox{<typeinfo>} deklariert ist.
Type\_info-Objekte können nicht kopiert werden, es kann höchstens
ihre Adresse in einem Zeiger abgespeichert werden. Man muss aber beachten, dass für denselben Typen durchaus mehrere Objekte im System existieren können. 
Man muss also immer die Objekte direkt miteinander vergleichen, nie
ihre Adressen. Ein sinnvoller Einsatz ist vorallem bei Vererbung und
Polymorphismus möglich, funktioniert aber nur, wenn die beteiligten Klassen
virtuelle Funktionen enthalten.
\begin{CCode}
#include <typeinfo>

class A {...};
class B : public A {...}

void f(A *p) {
   if (typeid(*p) == typeid(B) {
      ...
   }
}
\end{CCode}

In diesem type\_info-Objekt sind alle verfügbaren Informationen über die betroffene
Klasse abgelegt. Im System gibt es für jeden Datentyp und jede Klasse (Auch Benutzerdefinierte) ein (oder mehrere) entsprechende type\_info-Objekte.  

Der implementierungsunabhängige Teil der Klasse type\_info sieht wie folgt aus.
\begin{CCode}
class type_info 
   {
   public:
     virtual ~type_info();
 
     
     // Liefert einen Implememntationsabhaengigen String zurueck
     // (Ist meistens der Name der Klasse)
     const char* name() const;
 
     // Zum Sortieren von type_info-Objekten
     bool before(const type_info& __arg) const;
     
     // Pruefen, ob zwei type_info-Objekte gleich sind
     bool operator==(const type_info& __arg) const;
     bool operator!=(const type_info& __arg) const;
     
   private:
     // Kopieren und Zuweisen ist nicht erlaubt
     type_info& operator=(const type_info&);
     type_info(const type_info&);
   };
\end{CCode}
 
Zu Debug- und Testzwecken ist vorallem die Methode name() interessant.
Was genau die Funktion zurückliefert ist implementationsabhängig,
aber es muss in jedem Fall ein String sein, und man darf erwarten
das er in irgendeiner Form den Typen des betroffenen Objektes
enthält. Nachfolgend ein entsprechendes Beispiel: 
\begin{CCode}
#include <typeinfo>

class A {...};
class B : public A {...}

void f(A *p) {
   cout << "Objekt vom Typ <" << typeid(*p).name() << "> erhalten" << endl;
   ...
}
\end{CCode}
Man sollte aber typeid nicht dazu verwenden, objektabhängigen
Code zu erzeugen, dazu sind die virtuellen Funktionen da. In gewissen
speziellen Notfällen ist es manchmal notwendig, auf typeid
zurückzugreifen.

\section{Typumwandlungen (Casts)}
In \Cpp gibt es mehrere Möglichkeiten, um Typen ineinander umzuwandeln. Einerseits
der aus C 'geerbte' \OviIndex{Castoperator} (Z. B. (int) ), der aber nicht mehr benutzt
werden sollte, andererseits die neuen, flexibleren Castoperatoren dynamic\_cast<>(), static\_cast<>(), const\_cast<>() und reinterpret\_cast<>(). 


\paragraph*{reinterpret\_cast<>() und (cast)}
Mit dem Castoperator \OviIndex{reinterpret\_cast<>()} kann wie mit dem aus C bekannten (cast) eine \OviIndex{Typumwandlung} erzwungen werden. Der Castoperator besteht aus dem Schlüsselwort reinterpret\_cast und einem in spitzen Klammern eingeschlossenen gültigen Datentyp. Der Castoperator wird wie ein Funktionsaufruf benutzt und hat eine entsprechend hohe Priorität. Bei der Umwandlung wird versucht, denn Wert
der Daten soweit wie möglich zu erhalten, was aber oft nicht möglich ist, wenn ein grösserer Datentyp in einen kleineren gezwängt wird. Grundsätzlich kann aber mit dem Castoperator jeder skalare Typ (Zeiger und arithmetische Typen) in jeden anderen skalaren Typen umgewandelt werden.
\begin{CCode}
class D {...};

int a,    *ip;
float f,  *fp;
D     d,  *dp;

// Mit dem neuen reinterpret_cast<>()
a = reinterpret_cast<int>(f);     // OK
f = reinterpret_cast<float>(a);   // OK
d = reinterpret_cast<D>(a);       // Fehler, D ist nicht skalar
 
f  = reinterpret_cast<float>(pi); // OK, aber sinnlos, wandelt Adresse in float um
pi = reinterpret_cast<int *>(f);  // OK, aber sinnlos, wandelt float in Adresse um
dp = reinterpret_cast<D*>(ip);    // OK, Zeiger duerfen ineinander umgewandelt
                                  //     werden
dp = reinterpret_cast<D*>(f);     // OK, aber sinnlos, wandelt float in Adresse um

// Dasselbe mit dem (cast) aus C
a = (int)   f;  // OK
f = (float) a;  // OK
d = (D)     a;  // Fehler, D ist nicht skalar

f  = (float) pi; // OK, aber sinnlos, wandelt Adresse in float um
pi = (int *) f;  // OK, aber sinnlos, wandelt float in Adresse um
dp = (D*)    ip; // OK, Zeiger duerfen ineinander umgewandelt werden
dp = (D*)    f;  // OK, aber sinnlos, wandelt float in Adresse um
\end{CCode}

\paragraph*{static\_cast<>()}Mit dem Castoperator \OviIndex{static\_cast<>()} kann zwischen verwandten Typen gewandelt werden, also z. B. zwischen numerischen Typen wie int und float, oder zwischen verschiedenen Zeigertypen. Aber Zeiger in Numerische Werte zu konvertieren und umgekehrt ist nicht möglich. Diese Variante ist sicherer, und
dem reinterpret\_cast<>() wann immer möglich vorzuziehen.
\begin{CCode}
class D {...};

int a,    *ip;
float f,  *fp;
D     d,  *dp;

a = static_cast<int>(f);      // OK
f = static_cast<float>(a);   // OK
d = static_cast<D>(a);       // Fehler, D ist nicht skalar
 
f  = static_cast<float>(pi);  // Fehler, Zeiger nach Numerisch ist nicht moeglich
pi = static_cast<int *>(f);   // Fehler, Numerisch nach Zeiger ist nicht moeglich
dp = static_cast<D*>(ip);     // OK, Zeiger dürfen ineinander umgewandelt werden
dp = reinterpret_cast<D*>(f); // Fehler, Numerisch nach Zeiger ist nicht moeglich
\end{CCode}

\paragraph*{const\_cast<>()}Mit dem \OviIndex{Castoperator const\_cast<>()} kann die Konstantheit (const) und die Volatilität (volatile) entfernt werden. Weitergehende Umwandlungen sind nicht möglich.
\begin{CCode}
class D {...};

const int   ca;
      int   a;
const float cf;
      float f;
const D    *cdp;    // Zeiger auf konstantes D
      D    *dp;     // Zeiger auf D

dp = cdp;                     // Fehler, const-Zeiger an normalen Zeiger 
                              // zuzuweisen ist verboten
a  = const_cast<int>(f);      // Fehler, unterschiedliche Datentypen
f  = const_cast<float>(cf);   // OK, nur const entfernt
dp = const_cast<D*>(cdp);     // OK, nur const entfernt
\end{CCode}

\paragraph*{dynamic\_cast<>()}Mit dem Castoperator \OviIndex{dynamic\_cast<>()} können Zeiger und Referenzen auf Objekte innerhalb der Vererbungshierarchie ineinander umgewandelt werden. Dabei wird aber zu Laufzeit geprüft, ab das betroffene Objekt zum neuen Typ passt, sonst wird eine Exception geworfen (Referenzen) oder ein Nullzeiger geliefert (Zeiger). Das heisst für eine erfolgreiche Umwandlung muss das betroffene Objekt tatsächlich der gewünschten Klasse angehören, oder davon abgeleitet sein.
\begin{CCode}
class Y {...};
class A {...};
class B : public A {...};
class C : public A {...};
class D : public C {...};

A a;
B b;
C c;
D d;
Y y;

A *pa = &a;
B *pb = &b;
C *pc = &c;
D *pd = &d;
Y *py = &y;

A *p = &c;
pc = p;                     // Compiler-Fehler, Zeiger passen nicht
pc = dynamic_cast<C *>(p)   // OK, p zeigt tatsaechlich auf ein C
p  = &d;
pc = dynamic_cast<C *>(p)   // OK, p zeigt auf etwas von C abgeleitetes
p  = &b;
pc = dynamic_cast<C *>(p)   // Runtime-Fehler, p zeigt auf nichts C-aehnliches
                            // pc wird auf NULL gesetzt
pc = dynamic_cast<C *>(py); // Runtime-Fehler, py zeigt auf nichts C-aehnliches
                            // pc wird auf NULL gesetzt
\end{CCode}

dynamic\_cast<>() setzt Polymorphismus voraus, d.h. mindestens eine Funktion
von beiden beteiligten Klassen muss virtuell sein.
 
\section{Namensbereiche (Namespaces)}
\OviIndex{Namespaces} sind als zusätzliche Strukturierungselemente eingeführt worden. Damit kann man eine Gruppe von Funktionen, Variablen und Klassen in eine Einheit zusammenfassen.
Wenn man von zwei verschiedenen Herstellern Klassenbibliotheken verwendet, kann es vorkommen, das beide Bibliotheken Klassen mit demselben Namen besitzen, welche aber nicht dieselben Funktionen erfüllen. Es könnte z. B. in einer Graphikbibliothek eine Klasse Error und in einer Netzwerkbibliothek ebenfalls eine Klasse Error geben. Ohne Namespaces könnte man das Problem nur durch ändern einer der Bibliotheken umgehen, was aber nur möglich ist, wenn man den Sourcecode zu der Bibliothek besitzt. Andernfalls ist man auf die Mitarbeit des Bibliothekherstellers angewiesen oder muss die Bibliothek in eigene Bibliotheken einpacken. Mit Namespaces lässt sich das Problem elegant umgehen, indem jede Bibliothek ihren eigenen \OviIndex{Namensbereich} erhält:

In Bibliothek 1
\begin{CCode}
namespace graphic {
   void print(void);
   class A {
       int i;
   }
}

void graphic::print(void) {
   cout << "Hallo";
}
\end{CCode}

In Bibliothek 2
\begin{CCode}
namespace network {
   void print(void);
   class A {
       int i;
   }
}

void network ::print(void) {
   cout << "Hallo";
}
\end{CCode}

Beim Benutzer
\begin{CCode}
network::A AnObjectOfClassAfromLibraryNetwork;
graphic::A AnObjectOfClassAfromLibraryGraphic;

graphic::print();   // Call of print from graphic
print();  // Illegal, do not know which print to use
\end{CCode}

Wenn man nicht jedes mal den Namensbereich angeben will, kann die using Direktive verwendet werden:
\begin{CCode}
using graphic::print;  // Benutze print aus graphic wenn nichts anderes gesagt wird

print();               // print aus graphic wird benutzt
network::print();      // print aus network wird benutzt
\end{CCode}

 
Um alle Funktionen eines Namensbereichs sichtbar zu machen, verwendet man folgendes using:
\begin{CCode}
using namespace graphic;
\end{CCode}
Achtung, dadurch geht die ganze Trennung verloren und der Vorteil der Namespaces verschwindet.
In den Standardbibliotheken sind alle Member im Namensbereich std definiert, also muss man ohne using-Direktive 
\begin{CCode}
std::cout << "Hallo" << std::endl; 
\end{CCode}

schreiben. Dies ist unumgänglich wenn man die neuen Headerfiles einbindet (<iostream>) anstelle der veralteten <iostream.h>.

Fuer Namensbereiche können auch \OviIndex{Aliase} defineirt werden, d.h. man kann für einen bestehenden
Namensbereich einen neuen Namen einführen.

\begin{CCode}
namespace bfh = BernerFachhochschule;

bfh::Print(); // ist nun Aequivalent zu

BernerFachhochschule::Print();


\end{CCode}

Es können auch \OviIndex{anonyme Namensbereiche} definiert werden, alle in
einem anonymen Namensbereich definierten Elemente sind nur
innerhalb der definierenden Übersetzungseineheit (Source-Datei)
verfügbar. Entspricht in etwa dem Prinzip von static in C.   

\begin{CCode}

namespace {

   void f();
}

// f() kann ab hier verwendet werden, aber nicht
// von anderen Modulen benutzt werden

\end{CCode}
 
\section{Strings}
\OviIndex{Strings} werden in C++ in Form der Klasse string unterstützt. Diese Klasse
bietet einen komfortablen Umgang mit Zeichenketten. Unter anderem wird
das Erzeugen, Kopieren, Vergleichen und Manipulieren von Zeichenketten
vereinfacht.

Einige Beispiele:
\begin{CCode}

   string s1 = "Hallo";
   string s2 = s1 + " Welt";
   s1[1] = 'e';                 // s1 wird zu Hello
   if (s1 == s2) { };           // Vergleicht Strings
   std::cout << s2;             // Gibt s2 aus       

\end{CCode}

Es gibt noch viele weitere Funktionen zum Suchen von Strings innerhalb Strings, Einfügen

Zur Umwandlung zwischen Zahlen und Zeichenketten können die \OviIndex{StringStream} Klassen verwendet

   std::istringstream is(s);      // stringstream mit s initialisieren
   
   is >> i;                       // Zahl aus is lesen und i zuweisen
    

\end{CCode}

\chapter{Templates (Schablonen)}
Obwohl \OviIndex{Templates} eigentlich nur ein neues Schlüsselwort einführen, wird damit ein
mächtiges Werkzeug zur verfügung gestellt, welches fast wie eine eigene Sprache wirkt und
sehr komplexe, veralgemeinerte und typunabhängige Programmierung erlaubt. Mit Templates kann parametrisierbarer Code geschrieben werden, das heisst Code, der unabhängig von Datentypen ist. 

Der Benutzer eines Templates kann selbst angeben, für welchen Datentyp das Template Code erzeugen soll, er kann dabei auch Datentypen angeben, die bei der Erstellung eines Templates nicht bekannt waren.
Es können \OviIndex{Funktionstemplate} und \OviIndex{Klassentemplates} erzeugt werden. Achtung, Templates werden normalerweise vollständig in einer Headerdatei definiert, welche vom Anwender des Templates mittels \#include eingebunden werden muss.

In diesem Kapitel werden Templates nur kurz vorgestellt, eine umfassende Beschreibung von Templates kann in entsprechender Fachliteratur nachgeschlagen werden. Wer schon mal einen Blick in einen der Standardheader der STL (Z. B. vector) geworfen hat, weiss das Template-Programmierung sehr schnell zu schwer nachvollziehbarem Code führt. (Die Anwendung dagegen ist meist sehr einfach). Template-Parameter und Template-Spezialisierungen werden hier nur am Rande gestreift.

\section{Funktionstemplates}
Mit dem nachfolgenden Template wird eine generische Funktion zur
Bestimmung des Maximums von zwei Argumenten gleichen Typs definiert.
Dies funktioniert aber nur für Typen, welche den Operator > definiert haben.
\begin{CCode}
template <class T> 
T Maximum (T a, T b) 
{
   T Resultat;
   if (a > b) {
      Resultat = a;
   } else {
      Resultat = b;
   }
   return Resultat;
}
\end{CCode}

T ist ein frei wählbarer Name, der als Platzhalter für einen vom Benutzer anzugebenden Datentyp steht. Wenn der Benutzer zum Beispiel folgende Codezeile schreibt:
\begin{CCode}
int a;
int b;

b = Maximum< int > (a, 12);
\end{CCode}

Wird vom Compiler automatisch folgende Funktion erzeugt:
\begin{CCode}
int Maximum (int a, int b) 
{
   int Resultat;
   if (a > b) {
      Resultat = a;
   } else {
      Resultat = b;
   }
   return Resultat;
}
\end{CCode}

Überall wo im Template vorher der Parameter T stand, wird dieser durch den angegebenen Typ ersetzt. Möchte man nun auch das Maximum von zwei Fliesskommawerten bestimmen will, erledigt man das wie folgt:
\begin{CCode}
float x;
float y;

x = Maximum< float > (x, 12.0);
\end{CCode}

 
Es ist nicht einmal nötig, dem Compiler zu sagen, welches Template erzeugt werden soll, solange er es durch die Argumente selbst eindeutig ausfindig machen kann, wie z. B. hier:
\begin{CCode}
long x;
long y;

x = Maximum (x, y);  // Compiler erkennt selbst, dass hier long verwendet werden soll
\end{CCode}
Anstelle des Schlüsselwortes class kann übrigens auch das Schlüsselwort \OviIndex{typename} verwendet werden.

\subsection{\"Uberladen von Funktionstemplates}
Das Funktionstemplate Maximum aus dem vorherigen Kapitel kann noch
verbessert werden. Wenn man es auf Zeiger anwendet werden die
 Zeiger verglichen, was meist nicht das gewünschte
Verhalten ist. Besser wäre es, die Objekte zu vergleichen, auf die von den Zeigern verwiesen wird. Dies erreicht man durch \OviIndex{\"Uberladen des Templates}:

\begin{CCode}
template <class T>        
T *Maximum (T *a, T *b)  // gilt fuer alle Zeiger
{
   T* Resultat;
   if (*a > *b) {         // Zeiger dereferenzieren, also Objekte vergleichen 
      Resultat = a;
   } else {
      Resultat = b;
   }
   return Resultat;
}
\end{CCode}
Dieses Template wird jetzt immer aufgerufen, wenn zwei Zeiger
miteinander verglichen werden sollen. Natürlich gibt es immer noch ein
Problem, wenn Doppelzeiger (T**) oder Mehrfachzeiger (T**...**) im Spiel sind, dann müsste dieses Spiel weitergetrieben werden.
Oder man kann das ganze elegant mit rekursivem Templateeinsatz erledigen. Das Nachfolgende Template kann für beliebig verschachtelte Pointer benutzt werden
da es sich selbst rekursiv benutzt. Diese Anwendung ist sicher nicht sinnvoll,
soll aber das Prinzip zeigen. 
\begin{CCode}
template <class T>      
T *Maximum(T *a, T *b)  // gilt fuer alle Zeiger
{
   T *Resultat;
   if (Maximum(*a, *b) == *a) {  // Trick, wieder Maximum verwenden
      Resultat = a;              // Code noetig weil Maximum T zurueckliefert
   } else {                      // Wir aber T* zurueckliefern muessen
      Resultat = b;
   }
   return Resultat;
}
\end{CCode}
Für Strings (char *) ist eine weitere \"Uberladung erforderlich.
\begin{CCode}
const char* Maximum(const char *a, const char *b)  // gilt fuer alle Strings
{
   const char* Resultat;
   if (strcmp(a, b) > 0) {  // Normales Stringcompare verwenden
      Resultat = a;
   } else {
      Resultat = b;
   }
   return Resultat;
}
\end{CCode}

\section{Klassentemplates}
\index{Klassentemplates}
Es kann auch eine ganze Klasse parametrisiert werden:
\begin{CCode}
template<class T>
class Complex {
   T Real;
   T Imaginaer;

   public:

   Complex(T r, T i) : Real( r ), Imaginaer( i ) {};
   Complex(T r)        : Real( r ), Imaginaer( 0 ) {};
   Complex()             : Real( 0 ), Imaginaer( 0 ) {};

   Complex operator +(const Complex &s) {return Complex (Real+s.Real, 
                                                         Imaginaer+s.Imaginaer); };
   Complex operator -(const Complex &s) {return Complex(Real-s.Real, 
                                                         Imaginaer-s.Imaginaer); };
   T GetReal() {return Real;};
   T GetImaginaer() {return Imaginaer;};
};
\end{CCode}

Benutzung des Templates
\begin{CCode}
   int b;

   Complex<int> ca(1, 4);   // Variable ca und cb sind komplex mit Integerwerten
   Complex<int> cb(0, 0);

   cb = ca + cb;
   b = cb.GetReal();

   Complex<double> cda(1, 4); // Variable cda, cdb sind komplex mit Doublewerten
   Complex<double> cdb(0, 0);

   cdb = cda + cdb;
\end{CCode}

Falls die Klasse Complex auch den > operator definiert hätte, könnte das Maximum Template auch auf Complex Template Objekte angewendet werden:
\begin{CCode}
cb = Maximum< Complex<int>  > (ca, cb);
\end{CCode}

(Auf das Leerzeichen zwischen den beiden > > achten, sonst wäre es der Schiebeoperator !)

Es sind auch Templates mit mehreren verschiedenen Typen möglich:

\begin{CCode}
template< class T1, class T2>
class SpeziellKomplizierteKlasse {
   T1 Wert1;
   T2 Wert2;
   ...
   ...
   ...
};
\end{CCode}

\subsection{Parametrisierung von Templates mit Konstanten}
Templates können in ihrer Parameterliste auch konstante
Werte (int, float, Adressen, ...) enthalten, welche 
innerhalb des Templates benutzt werden können.
(Z. B. Arraygrenzen, Adressen von Funktionen,
 Abbruchkriterium für Iterationen)
 
Beispiel:
\begin{CCode}
template< class T1, int SIZE>
class SpeziellKomplizierteKlasse {
   T1 Wert1[SIZE]; /* Array vom Typ T und der Grösse SIZE */
   ...
   ...
   ...
};
\end{CCode}

\subsection{Default Templateparameter}
\index{Default Templateparameter}
Wenn ein Template oft für einen bestimmten Typ
benutzt wird, kann eine Defaultwert für diesen Parameter
vorgegeben werden:

\begin{CCode}
template< class T1, class T2 = int>
class SpeziellKomplizierteKlasse {
   T1 Wert1;
   T2 Wert2;
   ...
   ...
   ...
};


SpeziellKomplizierteKlasse<char>  C1;       // fuer T2 wird int genommen

SpeziellKomplizierteKlasse<char, float> C2; // fuer T2 wird float vorgegeben

\end{CCode}

Der Defaulttyp wird oft von den anderen Parametern des
Templates abhängig gemacht, was zu folgendem Code führt:

\begin{CCode}
template< class T1, class T2 = HelperClass<T1> >
class SpeziellKomplizierteKlasse {
   T1 Wert1;
   T2 Wert2;
   ...
   ...
   ...
};

\end{CCode}
HelperClass ist hier selbst ein Template, welches mit T1
parametrisiert wird.


\subsection{Spezialisierung von Templates}
Wenn bei einem Template für einen bestimmten Datentypen,
oder eine Gruppe von Typen anderer Code erzeugt werden
soll, erreicht man dies durch Spezialisierung des Templates.
Dabei gibt es zwei Arten der Spezialisierung: Komplette und
Partielle Spezialisierung. 

Die Spezialisierungen von Templates dürfen im Code erst
nach der Deklaration/Definition des generellen Templates
erfolgen. 

\subsection{Komplette Spezialisierung von Templates}
Bei der \OviIndex{kompletten Spezialisierung} wird das Template 
für einen ganz bestimmten Datentypen spezialisiert,
zum Bespiel fuer den Fall das Komplex mit int
erzeugt wird, und wir besonders schnelle Algorithmen
für diesen Fall kennen und auch anwenden wollen:

\begin{CCode}
template<>      // Diese Spezialisierung braucht keine
                // weiteren Parameter mehr
class Complex <int> {  // Dies ist die Spezialisierung fuer int
   int Real;
   int Imaginaer;

   public:

   Complex(int r, int i) : Real( r ), Imaginaer( i ) {};
   Complex(int r)        : Real( r ), Imaginaer( 0 ) {};
   Complex()             : Real( 0 ), Imaginaer( 0 ) {};

   Complex operator +(const Complex &s) {... besonders effiziente ... ; };
   Complex operator -(const Complex &s) {... Implementation fuer int Werte ... ; };
   int GetReal() {return Real;};
   int GetImaginaer() {return Imaginaer;};
};
\end{CCode}


Benutzung des Templates
\begin{CCode}
   int b;

   Complex<int> ca(1, 4);   // Variable ca und cb sind komplex mit Integerwerten
   Complex<int> cb(0, 0);   // hier wird die spezialisierte Version benutzt

   cb = ca + cb;
   b = cb.GetReal();

   Complex<double> cda(1, 4); // Variable cda, cdb sind komplex mit Doublewerten
   Complex<double> cdb(0, 0); // hier wird die allgemeine Version benutzt

   cdb = cda + cdb;
\end{CCode}


\subsection{Partielle Spezialisierung von Templates}

Bei der \OviIndex{partiellen Spezialisierung} wird das Template 
für eine Gruppe von Datentypen spezialisiert,
zum Beispiel für den Fall das Komplex mit Zeigern
erzeugt wird, und wir angemessenen Code für
diesen Fall erzeugen wollen:
(Achtung, soll nur als einfaches Beispiel dienen, hat keine
 sinnvolle Anwendung)
\begin{CCode}
template<class T>      
class Complex <T*> {  // Dies ist die Spezialisierung fuer Zeiger
   T *Real;           // Achtung T ist nicht Zeiger, T* ist Zeiger
   T *Imaginaer;

   public:

   Complex(T* r, T* i) : Real( r ), Imaginaer( i ) {};

   // Liefert normale Complex<T> zurueck, nicht Complex <T*>
   // also ohne Zeiger (Geht sonst nicht...)
   Complex<T> operator +(const Complex &s) {return Complex (*Real+*(s.Real), *Imaginaer+*(s.Imaginaer)); };
   Complex<T> operator -(const Complex &s) {return Complex (*Real-*(s.Real), *Imaginaer-*(s.Imaginaer)); };
   
   int GetReal() {return *Real;};
   int GetImaginaer() {return *Imaginaer;};
};
\end{CCode}

Benutzung des Templates
\begin{CCode}
   int a, b;

   Complex<int *> ca(&a, &b);  // Variable ca ist komplex mit Zeigern
   
   Complex<int>   cb(0, 0);    // hier wird die normale Version benutzt

   cb = ca + ca;              
   b = ca.GetReal();

\end{CCode}

Partielle Spezialisierungen werden im allgemeinen erst bei komplexeren
Anwendung zur Verhinderung von Codeduplikation benutzt. So kann jede
Liste aus Zeigern gleich behandelt werden, es muss nicht für jeden
Zeigertyp neuer Code erzeugt werden. Dies lässt sich durch eine
komplette Spezialisierung mit void-Zeiger, und partieller Spezialisierung für
alle anderen Zeigern erreichen, wobei die partielle Spezialisierung
auf die void-Spezialisierung zurückgreift. Damit wird nur Code 
für die void-Variante erzeugt. Sonst würde für jeden neuen Zeigertyp
erneut Code produziert.

Bei der Instantiierung eines Templates wird immer die am meisten
spezialisierte, noch passende Version verwendet.
Detailliertere Informationen zur \OviIndex{Templatespezialisierung} findet man in der entsprechenden
Fachliteratur.

\subsection{Template-Metaprogrammierung}
Hier noch ein kleines (einfaches) Beispiel, was mit "`Template-Magie"'
alles möglich ist.

Das nachfolgende Beispiel kann Binärzahlen in Dezimalzahlen
umwandeln, und das zu Compilezeit, d.h. die Binärzahl wird
zu Compilezeit umgewandelt. Es wird also \emph{kein} zusätzlicher
Code erzeugt oder Speicherplatz benötigt!

\begin{CCode}
/* Binary-to-Decimal as Template Metaprogram */

/* Allgemeiner Fall, rekursive Auflösung */
template< unsigned long N > struct Bin
  {
  enum { value = (N % 10) +                /* Hier ist die Magie:                 */
           2 * Bin< N / 10 > :: value } ;  /* Wert Einerstelle plus den           */
                                           /* doppelten Wert der hoeherwertigen   */
                                           /* Stellen, diese werden durch         */
                                           /* rekusive Templateinstantiation      */
                                           /* gewonnen (Rekusion endet bei N = 0) */           
  } ;

/* Spezialisierung fuer den Fall N = 0 */
struct Bin< 0 >
  {
  enum { value = 0 } ;   /* Einfach, wenn N = 0 ist auch Binaerwert 0 */
  } ;
\end{CCode}

Dieses Template könnte wie folgt angewendet werden:

\begin{CCode}
int a = Bin<1010110>::value;
cout << Bin<1101010>::value << endl;
int Array x[Bin<1001>::value][13];       /* Definiert 9*13 Array! */
\end{CCode}
Man beachte, das Ergebnis dieses Templates ist eine gewöhnliche
numerische Konstante, die überall im Code eingesetzt werden
kann, wo eine solche erwartet wird.


Das Template soll nur das Prinzip zeigen, es fehlen hier noch
Kontrollen und Fehlerhandling für illegale Binärzahlen wie
101037012 oder zu lange Zahlen. Aber auch diese Probleme
lassen sich mit entsprechenden Template-Tricks lösen.

\subsection{Templates und Vererbung}
Von Templateklassen kann wie von normalen Klassen abgeleitet werden,
und Templateklassen können von normalen Klassen oder Templateklassen
erben. Templateklassen verhalten sich bei der Vererbung
wie 'gewöhnliche' Klassen.





\section{Container, Algorithmen und Iteratoren der \Cpp - Standardbibliothek}
Die \OviIndex{STL} (\OviIndex{Standard Template Library}) war eine der ersten Anwendungen, die umfassend von Templates gebrauch machte, und zudem die Erweiterung und Verbesserung der Templatesyntax zur Folge hatte. Mittlerweile ist die STL in die \OviIndex{\Cpp - Standardbibliothek}
integriert worden. In diesem Kapitel wird vorallem auf die aus der STL übernommenen Komponenten der \Cpp - Standardbibliothek eingegangen. 

Die STL abstrahierte viele oft gebrauchte Elemente und Baublöcke der SW-Implementation, sie stellt universelle Templates für \OviIndex{Container}, \OviIndex{Iteratoren} und \OviIndex{Algorithmen} zur Verfügung.

\OviIndex{Container} sind eine Art universelle Behälter für Daten. Dazu gehören Listen, Vektoren und Mengen. Bei den Containern ist nicht vorgeschrieben wie sie implementiert sind, sondern nur welche Eigenschaften sie besitzen müssen. Ein Vektor muss z.B. effizienten wahlfreien (direkten) Zugriff auf seine Elemente erlauben, eine Liste hingegen muss effizientes Einfügen und Entfernen von Objekten ermöglichen. Eine Map wiederum ordnet zwei Elemente einander zu. Dabei dient eines als Schlüssel und das andere als Wert. Mit einem Schlüssel kann man jederzeit den dazugehörigen Wert aus der Map holen. Eine Map ist somit ein assoziativer Container und kann als kleine Datenbank betrachtet werden.

\OviIndex{Iteratoren} sind eine Art universelle Zeiger, mit denen durch Container iteriert werden kann. Sie können unabhängig vom Container benutzt werden, wobei aber jeder Container seine eigenen Iteratoren besitzt. Mit Iteratoren können auch Bereiche von Sequenzen definiert werden. (Einer zeigt auf den Beginn und ein anderer auf das Ende einer Sequenz). Iteratoren haben je nach Container zusätzliche Eigenschaften (Wie Pointerarithmetik und Direktzugriff). 

\OviIndex{Algorithmen} wiederum können auf Sequenzen und Container angewendet werden. Es gibt unter anderem folgende Algorithmen: Suchen, Sortieren, Doppelte oder mehrfache Einträge entfernen, Zufällig Mischen, Suchen und entfernen, Eine Operation auf alle Elemente anwenden, Alle Elemente zählen die eine Bedingung erfüllen, ...

In dieser Übersicht werden Stellvertretend kurz die Liste, der Vektor und die Map vorgestellt:
 

\subsection{Kurzübersicht zu Container list<> aus der \Cpp - Standardbibliothek}
Als Beispiel erstellen wir eine \OviIndex{Liste} von int's
\begin{CCode}

Val = MyList.front();    //Liefert den Wert des ersten Elementes zurueck

\end{CCode}

\subsection{Kurzübersicht zu Container vector<> aus der \Cpp - Standardbibliothek}
Als Beispiel erstellen wir ein Array von double's. Ein \OviIndex{Vektor} kann an seinem Ende wachsen und schrumpfen, es können also Elemente angehängt und entfernt werden. Zudem ist Wahlfreier Zugriff mit [] (Ohne Bereichsüberprüfung) und der Methode at() (Mit Bereichsüberprüfung) möglich.

#include<vector>

std::vector<double> MyVector; //Erstellen einer Liste von Integerwerten

sort(MyVector.begin(), MyVector.end()) //Universeller Sortieralgorythmus, funktioniert

\end{CCode}

\subsection{Kurzübersicht zu Container map<> aus der \Cpp - Standardbibliothek}
Die \OviIndex{map} ist ein \OviIndex{assozitiver Container}, das heisst er verwaltet Wertepaare.
Ein Wert ist der \OviIndex{Schlüssel} des Paares, und der andere Wert beinhaltet die
Daten, welche unter diesem Schlüssel abgelegt sind. Der Schlüssel kann wie
der Index bei einem gewöhnlichen Array betrachtet werden, mit dem
Schlüssel erhält man Zugriff auf die dazugehörigen Daten. Deshalb muss
jeder Schlüssel innerhalb einer map einmalig sein. Die Daten werden innerhalb der
map nach dem Schlüssel in aufsteigender Reihenfolge sortiert, und beim
Iterieren in dieser Reihenfolge abgearbeitet. Zur einfachen Benutzung
der map wurde der Indexoperator [] überladen, so dass der Schlüssel direkt als
Index verwendet werden kann. Wird ein nicht existierender Schlüssel
benutzt, erzeugt die map automatisch einen leeren Eintrag für diesen
Schlüssel und liefert eine Referenz darauf zurück.

Der Schlüssel muss ein Datentyp mit einer eindeutigen Ordnung sein, d. H.
die Relation '<' muss definiert und eindeutig sein. (Sonst könnten die
Schlüssel nicht sortiert werden).

Der Wert kann ein beliebiger Datentyp sein (Im Extremfall eine Liste
oder eine weitere map). 

Als Beispiel erstellen wir eine map mit Studenten und ihren Noten.

\begin{CCode}
#include<map>
#include<algorithm>
#include<string>

using namespace std;          // Um std::map, std::cout... usw zu vereinfachen 

map<string, double> MyMap;    //Erstellen einer Map von Studenten und Noten
  
void f(void) 
{  
   MyMap["Hans"]   = 5.5;        //Ein paar Eintraege...
   MyMap["Fritz"]  = 5.0;        
   MyMap["Franz"]  = 5.5;
   MyMap["Max"]    = 4.5;
   MyMap["Moritz"] = 4.5;
   MyMap["Max"]    = 5.5;        // Vorheriger Wert von Max wird ueberschrieben

   float Note = MyMap["Fritz"];        // Lesen ist auch moeglich

   map<string, double>::iterator ptr; //Definieren eines Iterators (Zeigers) in die Map

   ptr = MyMap.begin();          //Iterator auf erstes Element der Map zeigen lassen;

   while(ptr != MyMap.end()) {   // Fuer alle Elemente in der Map
    
      // Name und Note ausgeben (first und second sind durch map vorgegeben)
      cout << ptr->first << " hat die Note " << ptr->second << endl;   
      ptr++;
   }
}

\end{CCode}

Mit einer map ist es sehr einfach, Daten zu verwalten. Für einen bestimmten Schlüssel kann es aber nur einen Eintrag geben. Falls mehrere Einträge mit demselben Schlüssel möglich sein sollen,
muss eine Multimap verwendet werden.

Für die map gibt es noch weitere, hier nicht vorgestellte Operationen wie das Suchen von
Einträgen, hinzufügen von Einträgen oder löschen von Einträgen. (Der Indexoperator [] Sucht \emph{und} fügt hinzu, falls nicht gefunden)


\chapter{C++ in Embedded Systemen}
\section{Startup Code}

Im Vergleich zu C muss der Startupcode von  \Cpp wesentlich mehr Arbeit leisten. Bei
C müssen nur alle globalen und statischen Variablen vor der ersten Benutzung initialisiert
werden, dies wird meist durch ein kurzes Codestück vor dem Aufruf von main() erledigt.

Bei  \Cpp müssen zusätzlich die Konstruktoren von allen globalen und statischen Objekten
vor dem Aufruf von main() ausgeführt werden, und entsprechend deren Destruktoren wenn
das Programm durch Aufruf von exit()\footnote{Ein \Cpp Programm sollte nie mit exit() beendet werden, da so die Destruktoren der automatischen (Auf dem Stack angelegten) Objekte nicht aufgerufen werden. Anstelle von exit() sollte das Exceptionhandling, oder allenfalls Rückgabewerte von Funktionen benutzt werden, so dass ein \Cpp Programm immer via Beenden von main() terminiert.}  oder beenden von main() sein Ende erreicht. Dabei darf nicht vergessen werden, dass neben den vom Programmierer definierten globalen und statischen Objekte auch in der Standard-Library globale und statische Objekte vorhanden sind (z.B. cin, cerr und cout).


Dieser Initialisierungsablauf von \Cpp  ist jedoch sehr Compilerabhängig, bei einigen Compilern muss nur eine Runtimefunktion (z.B. \_\_init()) aufgerufen werden, bei anderen wird vom Cpompiler eine Tabelle mit Funktionszeigern generiert, die in dieser Tabelle aufgeführten Funktionen müssen vor dem Aufruf von main() durch den Startupcode aufgerufen werden.

Zudem muss normalerweise der Linker angewiesen werden, diese Initialisierungsdaten
aus den einzelnen Modulen zusammenzutragen und am richtigen Ort im Speicher abzulegen.

Nachfolgend als Beispiel, wie es beim Carme-Kit gemacht werden muss:

Initialisierungsfunktion (Muss vor Aufruf von main() aufgerufen werden)
\begin{CCode}

extern unsigned int _ctors_start, _ctors_end; // Vom Linker definierte Symbole, beinhalten 
                                              // die Startadresse und die Endadresse der
                                              // Konstruktorentabelle

/* _Init_Constructors
 *
 * Diese Funktion ruft alle in der Konstruktorentabelle enthaltenen Konstruktoren auf
 */


extern "C" void _Init_Constructors( void ); // Soll im C-Modus compiliert werden

void _Init_Constructors( void ) {

   unsigned int *ctor = &_ctors_start;      // Zeiger auf Tabellenstart setzen
  
   while (ctor < &_ctors_end) {             // Über alle Einträge interieren
      ((void (*)(void))*ctor)();            // Aktuelle Funktion aufrufen   
      ctor++;
   }
}
\end{CCode}

Analog müsste das Ganze nach dem Ende von main() für die Destruktoren gemacht werden, sofern die
Applikation überhaupt einmal beendet wird. (Wenn das Programm nur durch
Ausschalten des Gerätes 'beendet' wird, ist der Aufruf der Destruktoren
nicht notwendig).


Zusätzlich muss im Linkerscript das zusammenführen der Konstruktoren/Destruktorentabellen codiert werden: 
\begin{CCode}

		. = ALIGN(4);	/* advance location counter to the next 32bit boundary*/
  PROVIDE(_ctors_start = .);   
  KEEP(*(SORT(.ctors.*)))
  KEEP(*(.ctors))
  PROVIDE(_ctors_end = .);
  PROVIDE(_dtors_start_ = .);
  KEEP(*(SORT(.dtors.*)))
  KEEP(*(.dtors))
  PROVIDE(_dtors_end_ = .);/*		*(.ctors*)	*/			

\end{CCode}

\subsection{Member Funktionen}
Jeder Memberfunktion wird beim Aufruf ein zusätzliches, vor dem Programmierer verstecktes Argument übergeben, nämlich ein Zeiger auf das Objekt für welches die Methode aufgerufen wird. Innerhalb der Methode kann auf diesen Zeiger mit dem Schlüsselwort this zugegriffen werden. Für dieses zusätzliche Argument wird etwas mehr Code und Rechenzeit benötigt. (In C müsste man bei der Verwendung von Strukturen die Adresse der Struktur übergeben, somit eigentlich kein Unterschied zwischen C und \Cpp )
\begin{CCode}
	x.f(a, 12);
  // wird zu	
	f(&x, a, 12);

	p->f(a, 12);
  // wird zu	
	f(p, a, 12);

\end{CCode}
 
\subsection{Klassen}
Klassen entsprechen im Wesentlichen den Strukturen in C und benötigen auch nicht mehr Speicherplatz. Sobald eine Klasse virtuelle Funktionen besitzt oder erbt, erhält jedes Objekt dieser Klasse eine zusätzliche Pointer-Membervariable, welche auf die Tabelle mit den virtuellen Funktionen verweist. Alle Objekte werden also um die Grösse eines Pointers vergrössert. Bei Klassen mit nur wenigen Membern kann dies die Objektgrösse bis vervierfachen (Z.B bei nur einem char-Member). Zudem muss für die Klasse eine Tabelle mit Zeigern auf die virtuellen Funktionen angelegt werden.

Bei Mehrfachvererbung werden weitere versteckte Zeiger im Objekt angelegt, da es für jede Basisklasse eine eigene virtuelle Funktionstabelle benötigt. Zudem sind je nach Kontext Offsetkorrekturen des this-Pointers notwendig, welche zusätzliche Rechenzeit benötigen.

Bei virtueller Vererbung werden entweder nochmals weitere versteckte Zeiger im Objekt angelegt, oder es sind weitere Einträge in der virtuellen Funktionstabelle notwendig. Zudem ergibt sich beim Zugriff auf Member der virtuellen Basisklasse eine weitere Indirektion, was etwas zusätzliche Rechenzeit kostet. Zudem sind je nach Kontext Offsetkorrekturen des this-Pointers notwendig, welche weitere zusätzliche Rechenzeit benötigen. 


\subsection{Virtuelle Funktionen}
Beim Aufruf einer virtuellen Funktion soll die Funktion, welche zum effektiven Objekt gehört, aufgerufen werden. Das erfordert je nach Situation zusätzlichen Aufwand. 

Wenn eine virtuelle Funktion direkt auf ein Objekt angewendet wird, kann der Compiler bereits zur Compilezeit bestimmen, welches Objekt vorliegt, und den Funktionsaufruf direkt codieren. In diesem Fall entsteht kein zusätzlicher Laufzeitaufwand.

Wenn eine virtuelle Funktion auf einen Pointer oder eine Referenz auf ein Objekt angewendet wird, kann der Compiler zur Compilezeit nicht bestimmen, welches Objekt vorliegt, es muss zur Laufzeit entschieden werden, welches Objekt vorliegt.

Das setzt voraus, dass in jedem Objekt eine Information über das Objekt selbst abgelegt ist. Üblicherweise wird dazu an erster Stelle im Objekt ein zusätzlicher Pointer auf eine Tabelle abgelegt. Dieser Pointer wird vtblptr (Pointer auf die Virtuelle Funktionstabelle) genannt. 

Für jede Klasse mit virtuellen Funktionen wird einmal eine solche Tabelle angelegt, alle Objekte einer Klasse verweisen auf dieselbe Tabelle. (Bei Mehrfachvererbung sind mehrere Tabellen pro Klasse notwendig).

Ausgehend von folgendem Codefragment 
\begin{CCode}
	class Base {
	   int x;
		 int y;
		
	public:
		           Base();
		  virtual ~Base();
		
		  virtual void f1(int x);  
		  virtual void f2();  
		  virtual void f3();
              void g();				
	
	};
	
	class Derived : public Base {
      int z;
		
		
	public:
		           Derived();
		  virtual ~Derived();
		
		  virtual void f1(int x);  
		  virtual void f2();  
		  virtual void f4();  
              void h();				
	
	};
	
	
	Base    ObjA();
	Derived ObjB();

\end{CCode}

In den Tabellen der virtuellen Funktionen sind die Adressen der entsprechenden Methoden abgelegt. In einer abgeleiteten Klasse können neue Einträge hinzugefügt werden, aber das Layout und die Reihenfolge der Einträge der Basisklasse müssen erhalten bleiben. Bei überschriebenen Methoden werden an der entsprechenden Stelle in der Tabelle die Adressen der neuen Methoden abgelegt, bei geerbten Methoden werden die Adressen der Methoden der Basisklasse abgelegt, und bei neuen Methoden wird die Tabelle erweitert und die neuen Einträge hinten angefügt. 

Der Aufruf einer virtuellen Methode würde in diesem Fall zu folgendem Code führen:
\begin{CCode}

// C++ Code
Base *p = &ObjB;
p->f1(5);

// Effektiv ausgeführter Code
p->vtblptr->f1(p, 5)
\end{CCode}
Es wird also nicht direkt die Funktion f1() aufgerufen, sondern es wird über den vtblptr in der Tabelle der virtuellen Funktionen die Adresse der effektiven Funktion geholt, und diese wird ausgeführt. Der Aufruf einer virtuellen Funktion kostet also einen zusätzlichen Zugriff auf die Tabelle der virtuellen Funktionen. Ausser bei sehr kurzen Funktionen ist dieser Zugriff meist vernachlässigbar. Es kann auch durchaus sinnvoll sein, in einem Interrupthandler virtuelle Funktionen zu benutzen, insbesondere wenn dadurch ein switch() oder ein Vektortabellenzugriff im Handler eingespart werden kann.

Bei Mehrfachvererbung und bei virtueller Vererbung sind je nach Kontext weitere Indirektionen sowie Offsetkorrekturen des this-Pointers notwendig, welche zusätzliche Rechenzeit benötigen. 
 
 
\subsection{RTTI (Runtime Type Information)}
Für die Runtime-Type-Information werden zusätzliche Inforamtionen für polymorphe Klassen abgelegt. Diese Information kann in die virtuelle Funktionstabelle integriert werden, oder die virtuelle Funktionstabelle wird um einen Eintrag auf die Typeninformation erweitert. Auf jeden Fall wird für jede Klasse zusätzlicher Speicher für diese Informationen benötigt. Insbesondere muss auch die ganze Vererbungshierarchie abgebildet werden.

Ausser beim expliziten Zugriff auf die Typeninformation mit typeid() hat die RTTI keinen Einfluss auf das Laufzeitverhalten.

\subsection{Dynamic Cast}
Um einen Dynamic Cast durchführen zu können, muss zur Laufzeit überprüft werden, ab das betroffene Objekt tatsächlich vom gewünschten Typ, oder davon abgeleitet ist. Dazu müssen die RTTI-Informationen der Vererbungshierarchie entlang untersucht werden. Dies kann je nach Vererbungshierarchie unterschiedlich lange dauern. Somit sollte dieses Feature in zeitkritischen Bereichen oder bei strengen Anforderungen an den Determinismus (Vorhersagbares Zeitverhalten) nicht verwendet werden.

Wenn ein Dynamic Cast auf Referenzen angewendet wird, können zudem Exceptions ausgelöst werden, somit gilt zusätzlich das unter Exceptionshandling gesagte.
 
\subsection{Exceptionshandling}
Beim Exceptionhandling müssen sämtliche Objekte, welche im normalen Programmlauf vom Punkt des Auftretens einer Exception bis zum Punkt der passenden catch-Anweisung gelöscht worden wären durch den Exception-Mechanismus gelöscht werden. (Das gilt nur für automatisch erzeugte Objekte, für Objekte die mit new erzeugt wurden ist der Programmierer selbst verantwortlich).

Um dies zu erreichen, müsste zur Laufzeit ständig eine Liste nachgeführt werden, in welcher alle noch zu löschenden Objekte eingetragen sind. Da dies den normalen Programmlauf stark verlangsamen würde, gibt es andere Ansätze.

Eine Lösung, welche den normalen Programmlauf nicht beeinträchtigt, ist das Konstruieren einer grossen Tabelle, in welcher für jede Codezeile abgelegt ist, welche Objekte an dieser Stelle bereits erzeugt worden sind. Diese Tabelle wird vom Compiler für jede Funktion während des Compilierens erzeugt, und vom Exceptionhandling beim Auftreten einer Exception konsultiert um festzustellen, welche Objekte noch zu löschen sind. Diese Tabellen belegen zusätzlichen Speicher, bei Speicherknappheit muss möglicherweise auf das Exceptionhandling verzichten.

Das Behandeln einer Exception ist ein sehr aufwendiger Prozess, vom Auftreten der Exception ausgehend muss sich das Exceptionhandling durch die umschliessenden Try-Blocks nach aussen vorarbeiten, bis ein Try-Block mit passendem Catch gefunden wird, und dies über Funktionsaufrufe hinweg. Um einen passenden Catch zu finden, müssen zudem die RTTI Informationen durchsucht werden. Deshalb genügt das Exceptionhandling deterministischen Anforderungen normalerweise nicht.

Zudem darf eine Interruptroutine auf keinen Fall mit einer Exception beendet werden (Der Interrupt kann den normalen Programmfluss ja irgendwo Unterbrechen), wegen des nicht derterministischen Verhalten sollte sogar ganz auf Exceptions in Interruptroutinen verzichtet werden.

\subsection{Inline Funktionen}
Bei Inlinefunktionen wird an der Stelle des Funktionsaufrufes der Code der Funktion eingefügt, und nicht die Funktion aufgerufen. Dies spart den Code für die Parameterübergabe und den Funktionsaufruf, und erlaubt weitergehende Optimierungen. 
Grundsätzlich spart man sich die Zeit für den Funktionsaufruf, wenn jedoch die Funktionsdauer gegenüber der Zeit für den Funktionsaufruf dominiert ist der Gewinn klein.

Auf der Gegenseite wird die Codegrösse beim Benutzen von Inlinefunktionen grösser. (Der Code für die Funktion wird bei jedem Funktionsaufruf erneut eingefügt. Wenn der Funktionscode mehr Platz belegt, als durch den Wegfall des Codes für den Funktionsaufruf eingespart wird, hat das mehr Platzbedarf für den Gesamtcode zur Folge). Bei Inlinefunktionen muss man also immer zwischen Geschwindigkeitsgewinn und Codezuwachs abwägen. Einzig bei ganz kurzen Funktionen (1-3 Zeilen) kann auch die Codegrösse abnehmen.

Wenn Inlinefunktionen über Pointer oder als virtuelle Funktionen eingesetzt werden, findet kein Inlining statt (Aber sehr wohl an Stellen bei der die Funktion direkt aufgerufen wird) 

\subsection{Templates}
Ein gut geschriebenes Template sollte nicht mehr Code als handgeschriebener Code erzeugen. Man muss aber beachten, dass ein einfacher Templateaufruf möglicherweise im Hintergrund zum Instanzieren vieler Templates mit dem entsprechenden Code führt. Zudem sind noch nicht alle Linker in der Lage, redundanten Code wieder zu entfernen, wenn dasselbe Template in verschiedenenen Modulen verwendet wird. Grundsätzlich spricht ejdoch nichts gegen den Einsatz von gut geschriebenen Templates, man sollte einfach den Speicherverbrauch im Auge behalten.

\subsection{Temporäre Objekte (Temporaries)}
Temoräre Objekte werden vom Compiler automatisch erzeugt, um Zwischenergebnisse aufzunehmen. Es handelt sich um Objekte, welche bei der Auswertung eines Ausdruckes automatisch erzeugt und wieder gelöscht werden. Auch Parameter und Rückgabewerte von Funktionen erzeugen oft temporäre Objekte. Optimierende Compiler versuchen, Temporaries soweit wie möglich zu vermeiden. Als Programmierer kann man Temporaries vermeiden, wenn man Referenzen statt die Objekte selbst verwendet.


Beispiel
\begin{CCode}
// Folgender Ausdruck
x = a + b + c;

// wird zu
x.operator=(operator+(operator+(a, b), c));

// und dies zu
t1 = operator(a, b);    // t1 ist temporaeres Objekt
t2 = operator(t1, c);   // t2 ist temporaeres Objekt
x.operator=(t2);
\end{CCode}
Nebst den beiden Temporaries t1 und t2 werden für die Ergebnisrückgabe der Operatorfunktionen weitere Temporaries erzeugt und wieder vernichtet.

Sobald Temporäre Objekte Nicht-Triviale Konstruktoren haben, muss jedes Temporäre Objekt konstruiert und destruiert werden, was viel Laufzeit kosten kann.

\section{Interrupt und Multithreading}

Wenn Interrupts oder Multithreading ins Spiel kommen, sind einige Punkte zu beachten:

Innerhalb von interruptroutinen sollte grundsätzlich auf Exceptions verzichtet werden, und sonst auf jeden Fall sicherstellen dass \emph{jede} Exception aufgefangen wird, befor das Ende der Interruptroutine erreicht wird, ansonsten wird der Code unweigerlich abstürzen.

Auch das Erzeugen von Objekten sollte innerhalb von Interrupts unterlassen werden, da auf jeden Fall die Konstruktoren (Rechenzeit) ausgeführt werden, und bei dynamischer Speicherverwaltung zusätzlich noch Speicher alloziert werden muss. Ebenso ist das Löschen von Objekten zu vermeiden.

Virtuelle Funktionen können innerhalb von Interruptreoutinen verwendet werden, man muss sich einfach bewusst sein, dass diese zusätzliche Indirektion Rechenzeit kostet. (Aber möglicherweise einen Switch() oder einen Callback erspart. Um virtuelle Funktionen einsetzen zu können, muss aber zuerst von der (statischen) Interuptfunktion eine Methode von einem Objekt aufgerufen werden.
(Eine Interuptfunktion kann nur eine globale Funktion oder eine statische Methode sein, da diese von der Hardware, ohne Kenntnisse eines Objektes, aufgerufen wird.) 

In Multithreadingumgebungen muss Sichergestellt werden, dass Konstruktoren, Destruktoren und Memorymanagement Threadsafe sind, ansonsten dürfen innerhalb von Threads keine neuen Objekte erzeugt werden. Ein Thread darf zudem \emph{nie} mit einer Exception beendet werden (ausser das verwendete OS kann mit Exceptions umgehen). Vor dem Threadende müssen somit immer alle Exceptions aufgefangen werden (Den gesammten Thread-Code in Try-Catch einschliessen.). 

\section{Persistenz}
Wenn Objekte ihren Zustand bei einem Stromausfall (Ausschalten) behalten sollen, spricht man von Persistenz. Dazu kann man Konstruktoren entwerfen, welche ihre Daten aus einem Dateisystem, FLASH-Speicher oder Akku-gepuffertem RAM holen, und entsprechende Funktionen, welche dieses Daten bei Bedarf auf diese nichtflüchtigen Medien speichern. Placement new() oder eigene new/delete Operatoren können auch benutzt werden, um Objekte in persistentem Speicher zu erzeugen, wobei bei jedem Systemstart geprüft werden muss, ob der Speicherinhalt noch gültig ist, bevor er wieder verwendet wird.

\section{Neuerungen}
\subsection{RValue Referenzen und Move Konstruktor / Move Zuweisung}
\label{subsec:RValueRefs}

RValue Referenzen werden durch T \&\&r (im Gegensatz zu T \&r bei gewöhnlichen Referenzen) definiert. Beim Kopieren und Übergeben von Parametern werden in \Cpp oft temporäre
Objekte erzeugt, welche jedoch 'Schreibgeschützt' sind. 

Mit RValue Referenzen können nun Temporäre Objekte verändert werden. Dies wird benötigt, um den neuen Move-Konstruktor zu implementieren. Beim Move-Konstruktor wird der Inhalt des Quellobjektes in das Zielobjekt verschoben, anstelle kopiert wie beim Kopierkonstruktor. Damit kann überflüssiges Kopieren unterbunden werden. (Da ein Temporäres Objekt sowieso gelöscht wird, stört es nicht, wenn sein Inhalt bereits vor dem Löschen entfernt wird. Man kann deshalb mittels Move-Konstruktor direkt den Inhalt dieses Temporär-Objektes in das Neue Objekt verschieben, und spart sich so einen Kopiervorgang. Man mus nur dafür sorgen, dass der Destruktor des Temporären Objektes die 'Entwendeten' Daten nicht löscht).

Beispiel:
\begin{CCode}


class Demo {
		// ...
		Demo(const Demo &Src);            // Kopierkonstruktor
		Demo(Demo&&Src);                  // Move-Konstruktor
		Demo& operator=(const Demo &Src); // Zuweisungsoperator (Kopierend)
		Demo& operator=(Demo &&Src);      // Zuweisungsoperator (Schiebend)
	};	 
 
\end{CCode}

\subsection{constexpr}
Mit dem Schlüsselwort constexpr können Ausdrücke und Rückgabewerte von Funktionen als
Konstant deklariert werden. Damit werden solche Ausdrücke und Funktionen auch an Stellen akzeptiert, an denen nur Konstanten zulässig sind, und der Ausdruck/Funktionsaufruf wird bereits zur Compilezeit ausgewertet. constexpr-Funktionen dürfen neben der return-Anweisung keine weiteren Anweisungen mehr enthalten.

Beispiel
\begin{CCode}

constexpr int GetLength() 
{
   return 15;
}
 
int AnArray[GetLength() + 3]; // Ok, Arraygrösse ist konstant

constexpr int MakeDouble(int x) 
{
   return x * 2;
}

int AnotherArray[MakeDouble(8) + 3]; // Ok, Arraygrösse ist konstant


constexpr double Erdanziehung = 9.81;
constexpr double MondAnziehung = Erdanziehung / 6.0;
\end{CCode}

Entsprechend können auch Konstruktoren als constexpr definiert werden, dies erlaubt das Erzeugen konstanter Objekte. constexpr-Funktionen können auch mit Nicht-Konstanten Argumenten aufgerufen werden, in diesem Fall verliert ihr Rückgabewert für diesen Aufruf seine Konstantheit.

Mit constexpr kann insbesondere auch in Embedded Umgebungen effizienter, maschinenennaher und typsicherer Code erzeugt werden, da Berechnungen soweit möglich bereits vom Compiler vorgenommen werden. Zusammen mit Inline-Methoden eröffnen sich so neue Optimierungsmöglichleiten.

\subsection{extern Templates}
Mit dem Schlüsselwort extern kann das Instantiieren eines Templates unterdrückt werden. Damit kann das Template benutzt werden, ohne dass der Compiler im aktuellen Modul dafür Code erzeugt. Bisher wurde in jedem Modul, welches ein Template benutzte, auch der Code dafür erzeugt. Dies hatte viel redundanten Code zur Folge, welcher meist vom Linker wieder entfernt wurde. Mit extern kann das Erzeugen gleich von vornherein unterbunden werden. Selbstverständlich muss das Template minsdestens in einem Modul des Projektes weiterhin instantiiert werden. (Analog zu globalen Variablen und Funktionen)


\begin{CCode}
extern template class std::vector<DemoClass>; // Template darf in diesem Modul benutzt werden, der Code wird aber woanders erzeugt.
\end{CCode}

\subsection{Initialisiererlisten}
Neu können Objekte analog Arrays und Strukturen mit Listen initialisiert werden. Dazu mus jedoch ein entsprechender Konstruktor definiert werden, der als Argument ein Template vom Typ std::initializer\_list entgegennimmt.

Beispiel:
\begin{CCode}

class DemoClass {
   public:
   DemoClass(std::initializer_list<int> list);
};


DemoClass TheObject = {1, 4, 5, 6}; // Calls initializer-list constructor

\end{CCode}

Das Template std::initializer\_list kann aber auch bei gewöhnlichen Funktionen benutzt werden, um einer Funktion direkt in geschweiften Klammern eingeschlossene Listen zu übergeben. Container aus der Standardlibrary können ebenfalls direkt mit Initialisiererlisten Initialisiert werden.

\subsection{Vereinheitlichte Initialisierung}
Ein Objekt einer Klasse kann neu auch wie eine Struktur mittels gescheiften Klammern initialisiert werden. Sofern kein Initialisiererlisten-Konstruktor existiert, wird der
passende Konstruktor aufgerufen (falls vorhanden).

Beispiel:
\begin{CCode}

class DemoClass {

	 private:
	 int a;
	 float b;	

   public:
   DemoClass(int _a, float _b) : a(_a), b(_b) {};
	 
};


DemoClass TheObject = {10, 3.14f}; // Ruft Konstruktor auf
DemoClass TheObject{10, 3.14f};    // Dito, aber ohne =

// Ebenfalls erlaubt in Return-Statements und bei Funktionsaufrufen

DemoClass f() {
   return {2, 2.7f};
}

void f2(DemoClass c) {
}

f2({5, 1.23f});


\end{CCode}

\subsection{Automatische Typerkennung}
Mit dem Schlüselwort auto wird der benötigte Datentyp einer Variable vom Compiler automatisch bestimmt. Dazu wird anstelle des Datentypen bei der Variablendefinition das Schlüsselwort auto benutzt. Mit dem Schlüsselwort decltype kann eine Variable vom Typ des angegeben Ausdrucks definiert werden:
\begin{CCode}

auto Smart = FunktionMitKompliziertemRueckgabetyp(....);  // Smart erhält automatisch den passenden Typen

auto Magic = 5;  // Magic erhält den Typ int


decltype(Magic) OtherMagicVariable = 0; // Selber Typ wie Magic

for (auto itr = myvec.cbegin(); itr != myvec.cend(); ++itr) {
   ...
}




\end{CCode}

\subsection{Automatische For-Schlaufe}

Eine For-Schlaufe, welche über alle Elemente eines Arrays, einer Initialisiererliste oder eines Standard-Containers iterieren soll, kann wie folgt vereinfacht werden:

\begin{CCode}
int Elemente[5] = {1, 2, 3, 4, 5};
int Summe = 0;

// Ueber alle Elemente iterieren
for (int &r : Elemente) {
    Summe += r;
}

\end{CCode}

Dies funktioniert für alle Objekte, welche begin() und end() definiert haben, wobei begin() und end()  Iteratoren zurückliefern müssen.

\subsection{Anonyme Funktionen (Lambda Funktionen)}

Lambda Funktionen sind Funktionen, welche direkt anstelle eines Funktionspointers definiert und übergeben werden. Hilfreich bei Vergleichsfunktionen und ähnlichem.

Grundsätzlich wird eine Lambda-Funktion wie folgt definiert:
\begin{CCode}
[capture](parameters)->return_type{body}
\end{CCode}

Mit \emph{Capture} kann der Funktion Zugriff auf Variablen der definierenden Funktion (Welche nicht mit der aufrufenden Funktion uebereinstimmen muss) gegeben werden. Der Zugriff kann by Value oder by Reference erfolgen. Die Capture-Liste darf auch leer sein. Achtung, eine Lambda-Funktion mit einem Capture-Block darf nicht aufgerufen werden, wenn die referenzierten Variablen nicht mehr existieren.

\emph{Parameter} ist die Parameterliste der Funktion (Wie bei gewöhnlichen Funktionen). Hier kann der Aufrufer Argumente an die Funktion übergeben. Bei einer Parameterlosen Funktion darf das (leere) Klammerpaar auch weggelassen werden.

\emph{-\textgreater return\_type} definiert den Typ des Rückgabewertes, dieser Teil kann weggelassen werden, dann wird der Rückgabetyp durch das return-Statement definiert oder void wenn kein return-Statement vorhanden ist. 

\emph{Body} enthält den Code der Funktion.

Beispiele
\begin{CCode}

[](int x, int y) -> float { return x + y; } // Rückgabetyp ist float
[](int x, int y) { return x + y; } // Impliziter Rückgabetyp durch return
[](int& x) { ++x; }                // Rückgabetyp ist void (Weder return noch ->)
[]() { ++global_x; }               // Keine Argumente, zugriff auf globale Variable
[]{ ++global_x; }                  // Dasselbe, () kann weggelassen werden
[&q, r](int& x) { q += r*++x; }    // Zugriff auf die Variablen q und r der
                                   // definierenden Funktion erlauben
// Einsatzbeispiel
std::vector<int> Liste;
int Total = 0;
for (int i=0; i<5; ++i) {
   Liste.push_back(i);
}

// for_each wird als drittes Argument eine Lambda-Funktion (Eigentlich ein
// Zeiger darauf) übergeben.  
// for_each ruft fuer jedes Element in der Liste diese Lambda-Funktion
// auf, das aktuelle Element wird jeweils in x uebergeben. Der lambda-Funktion 
// wird zudem der Zugriff auf die Variable Total erlaubt.  

std::for_each( begin(Liste), end(Liste), [&Total](int x) {
                                            Total += x;
                                         }
						 );
\end{CCode}
Lambda-Funktionen können auch in Variablen (Entsprechend auch in Arrays und anderen
Containern) abgespeichert werden. Dabei erleichtert die automatische Typerkennung mit den Schlüsselworten auto oder decltype die Variablendefinition.

\subsection{Alternative Funktionsdefinition}

Der Rückgabetyp von Funktionen kann neu auch nach der Argumentenliste angegeben werden. Dies ist vorallem bei komplexeren Templates von Vorteil.

\begin{CCode}

// Der Rückgabetyp ist, was auch immer der Ergebnistyp der Addition von x und y ist
template<class T1, class T2>
  auto MagicFunction(const T1 &x, const T2 &y) -> decltype(x+y) {return x + y;}

auto f1(int x, int y) -> float // Rueckgabewert ist float
{
}

class Demo {
    auto MemberFunktion(int x, int y) -> int; // Rueckgabewert ist int
};
 
auto Demo::MemberFunktion(int x, int y) -> int // Rueckgabewert ist int
{
    return x + y;
}

\end{CCode}


\subsection{Vereinfachte Objektkonstruktion}

Neu dürfen Konstruktoren andere Konstruktoren der selben Klasse aufrufen, und es können Konstruktoren der Basisklasse übernommen werden. Zudem können Defaultwerte für Membervariablen festgelegt werden.

\begin{CCode}
class Demo  {
    int x;
 
public:
    Demo(int _x) : x(_x) {}
    Demo() : Demo(42) {}    // Ruft bestehenden Konstruktor auf
};


class Example : public Demo {

    using Demo::Demo; // Konstruktoren von Demo uebernehmen
};


class Test {
public:

    Test(int _x) : x(_x) {} // Setzt x auf den uebergebenen Wert 
 
private:

		            // im Konstruktor etwas anderes definiert
		
}\end{CCode}


\subsection{Kontrolliertes Überschreiben von Funktionen}

Mit den Bezeichnern \index{override}\emph{override} und \index{final}\emph{final} kann das versehentliche überschreiben / nicht-überschreiben von Methoden verhindert werden. Diese beiden Bezeichner sind keine Schlüsselworte, d.h. sie dürften weiterhin als Namen verwendet werden. Sie haben nur innerhalb von Methodendeklarationen eine spezielle Bedeutung. 

Methoden mit dem Bezeichner override müssen eine Methode der Basisklasse überschrieben, sonst wird eine Fehlermeldung ausgegeben, Methoden mit dem Bezeichner final dürfen von keiner abgeleiteten Klasse überschrieben werden.

\begin{CCode}
class Basisklasse {
    public:
    virtual void ModifizierbareFunktion(float);
    virtual void NichtWeiterUeberschreibbareFunktion(float) final;
};
 
class Abgeleitet : Basisklasse {
    public:
    virtual void ModifizierbareFunktion(int) override; // Fehler, keine passende Methode in 
		                                                   // der Basisklasse (int statt float verwendet)
    virtual void NichtWeiterUeberschreibbareFunktion(float);// Fehler, darf nicht ueberschrieben werden
		
};


\end{CCode}

\subsection{Nullpointer}
Neu gibt es ein Schlüsselwort für den Nullpointer: nullptr
Damit kann zwischen dem Integerwert 0 und dem Nullzeiger unterschieden werden. Dieser Unterschied ist speziell bei überladenen Funktionen wichtig.

\subsection{Typsicherere Enummerationen}
Neu gibt es einen typsicheren Aufzählungstypen. Dieser kann nicht mehr mit andern Datentypen wie int, aber auch nicht mit anderen Aufzählungstypen gemischt werden. Dafür kann der unterliegende Datentyp (Speicherbedarf fuer einen Enum-Wert) angegeben werden .

Zudem sind die einzelnen Enumwerte im Namensbereich des jeweiligen Enums, es können also
mehrere Enums dieselben Bezeichner verwenden.   

Neue Syntax:
\begin{CCode}

enum class Aufzaehlung {
    Item1,
    Item2,
    Item3 = 100,
    Item4         // Erhaelt den Wert 101 
};

int i = Aufzaehlung::Item2;          // Compilerfehler

Aufzaehlung a = Item1;               // Compilerfehler
Aufzaehlung a = Aufzaehlung::Item1;  // OK
a = 12;                              // Compilerfehler

if (a == 17)                         // Compilerfehler
 
// Angabe der gewuenschte Datengrösse
enum class Enum2 : unsigned int {Item1, Item2};

\end{CCode}

\subsection{Template-Verschachtelung $>>$}
Bei verschachtelten Templates wird $>>$ als schliessendes Klammerpaar zweier verschachtelter Templates angesehen (Solange dies Sinn ergibt), und nicht mehr als Schiebeoperator. Sollte das nicht erwünscht sein, müssen Klammern gesetzt werden.

\subsection{Explizite Umwandlungsoperatoren}
Neu können neben Konstruktoren auch Umwandlungsoperatoren mit dem Schlüsselwort 'explizit' versehen werden. Damit werden diese vom Compiler nicht mehr automatisch aufgerufen, sondern müssen explizit vom Programmierer aufgerufen werden. (Die Umwandlung nach bool in Bedingungen bei Kontrollstrukturen und für Argumente fuer Logische Operatoren gilt als expliziter Einsatz)

\subsection{Template mit variabler Argumentenanzahl}

Neu können Templates Variable Argumentenlisten haben.
\begin{CCode}
template<typename... Values> class tuple;
\end{CCode}

\subsection{Erweiterte Stringliterale}
Neu können auch UTF-Codierte Stringliterale definiert werden, dazu werden die Prefixes u8, u und U verwendet.

\begin{CCode}
u8"Ein UTF-8 String"  // hat den Typ const char[]
u"Ein UTF-16 String"  // hat den Typ const char16_t[]
U"Ein UTF-32 String"  // hat den Typ const char32_t[]
\end{CCode}

Unicode-Zeichen können im String auch direkt mit ihrem Code angegeben werden:
\begin{CCode}
u8"Ein spezielles UTF-8 Zeichen: \u2018" 
u"Ein spezielles UTF-16 Zeichen: \u2018" 
U"Ein spezielles UTF-32 Zeichen: \U00002018"  
\end{CCode}

Zudem können Escape-Freie Raw-Strings definiert werden. Diese werden mit dem Prefix R gekennzeichenet, der eigentliche String wird mit "'\emph{Begrenzung}( und )\emph{Begrenzung}"' eingeschlossen, der eingeschlossene Text kann alle Zeichen (Inklusive ()"'\textbackslash) enthalten. \emph{Begrenzung} kann irgend eine Zeichenfolge (Auch eine Leere) mit maximal 16 Zeichen sein.

Beispiel:
\begin{CCode}
R"(Ein "Muehsamer" String mit \ "komischen" \ Zeichen)"

// Folgender String endet erst bei )XYZxyz" und darf deshalb auch )" enthalten
R"XYZxyz(Ein "( "Konfuser" )xyz" String mit )" gemeinen )" Zeichen)XYZxyz" 
\end{CCode}

R und U können auch kombiniert werden.

\subsection{Benutzerdefinierte Literale}

Bisher konnten nur die vom Compiler definierten Literale benutzt werden, neu kann der Anwender eigene Suffixes für Literale definieren. Also anstelle 123u könnte jetzt auch 123\_Magic oder 123\_kg benutzt werden. Ein benutzerdefinierter Suffix muss immer mit einem \_ beginnen (Literale ohne \_ am Anfang sind reserviert für zukünftige Erweiterungen).

Numerische Literale können dabei als Strings, als unsigned long long oder als long double Werte verarbeitet werden. Dazu wird eine neue Operatorfunktion eingeführt, welche entsprechend überladen werden muss:

\begin{CCode}
// Erhaelt einen String mit der Zeichenkette vor dem Suffix
OutputType operator "" _Magic(const char * literal_string); 

OutputType operator "" _kg(unsigned long long); // Erhaelt den Integerwert des Literals
OutputType operator "" _rad(long double);       // Erhaelt den Fliesskommawert des Literals
\end{CCode}

Die Operatorfunktionen können dabei einen beliebigen Rückgabewert liefern, im Allgemeinen sollte ein passendes Objekt erzeugt werden. (Der Suffix \_kg könnte zum Beispiel ein Objekt der Klasse Gewicht liefern).

Für Stringliterale gibt es 4 verschiedene Operatorfunktionen, je nach Literaltyp:

\begin{CCode}
OutputType operator "" _suffix(const char     * StringValues, size_t StringLaenge);
OutputType operator "" _suffix(const wchar_t  * StringValues, size_t StringLaenge);
OutputType operator "" _suffix(const char16_t * StringValues, size_t StringLaenge);
OutputType operator "" _suffix(const char32_t * StringValues, size_t StringLaenge);
\end{CCode}

Stringvalues zeigt auf die Zeichenfolge des Literals, StringLaenge gibt die Anzahl der Zeichen im Literal an.

\subsection{Explizites Erlauben oder verbieten automatisch erzeugter Memberfunktionen}
Diverse spezielle Methoden wie Zuweisungsoperator, Konstruktor, new, ... werden vom Compiler automatisch erzeugt. Neu kann der Programmierer einfacher darauf Einfluss nehmen:

\begin{CCode}
class Demo {
    Demo() = default;                       // Den Defaultkonstruktor anfordern.
    Demo(const Demo&) = delete;             // Den Kopierkonstruktor verbieten.
    Demo & operator=(const Demo&) = delete; // Den Zuweisungsoperator verbieten.
    void f(double i);                    
    void f(int) = delete;                   // f darf nicht mit int aufgerufen 
                                            // werden
    template<class T> void f(T) = delete;   // f mit irgend etwas anderem als 
		                                        // double aufrufen ist verboten
};

\end{CCode}
Ohne die beiden letzten delete könnte f auch mit int aufgerufen werden, int würde in diesem Fall in double umgewandelt werden.

\section{Zusammenarbeit von C und \Cpp}
In \Cpp können Funktionen überladen werden und Elemente in Namensbereichen abgelegt werden. 
Um überladene Funktionen zu unterscheiden werden die Funktionsnamen vom \Cpp Compiler um die Argumentlistenbeschreibung sowie den Namensbereichen erweitert,
dies wird als \OviIndex{Name mangling} bezeichnet.
Der C Compiler hingegen benutzt direkt die Funktionsnamen. Für den Linker sind deshalb C und  \Cpp Module nicht kompatibel. 

Die Funktion int TuWas(int a, float Zahl, char *Text) könnte
in  \Cpp zum Beispiel als \_Z5TuWasifpc codiert werden.

Um dieses  \OviIndex{mangling} zu unterdrücken, kann in  \Cpp 
der Deklaration ein \emph{extern ''C''} vorangestellt werden.


Will man innerhalb eines Projektes Module mit C Code und Module mit
\Cpp Code mischen, müssen im \Cpp Teil Funktionen 
mit dem Zusatz \emph{extern ''C''} deklariert werden, sonst können sie
von C aus nicht aufgerufen werden. Auch die Prototypen der
C Funktionen müssen mit \emph{extern ''C''} deklariert werden, sonst
können sie vom \Cpp Teil nicht aufgerufen werden.

(Also alle Funktionen die von C und \Cpp gemeinsam benutzt
werden, müssen für den \Cpp Compiler mit \emph{extern ''C''}
deklariert werden, hingegen darf der C Compiler \emph{kein}
extern ''C'' sehen, da er dies nicht kennt.)

Um trotzdem nicht zwei verschiedene Headerdateien für
C und \Cpp zu haben, setzt man bedingte Compilierung ein,
dazu wird das vom \Cpp Compiler standardmässig definierte Makro \_\_cplusplus zur
bedingten Compilierung (Mittles \#ifdef) benutzt.
 

Gewöhnliche Strukturen (Ohne Methoden und Vererbung) 
und globale Nicht-Klassen-Variablen
können von C und  \Cpp ohne weiteres gemeinsam benutzt werden.

Objekte und Klassen hingegen können nicht direkt zwischen
C und \Cpp ausgetauscht werden (Zumindest nicht auf eine
portable Art).

Methoden können somit von C aus nicht aufgerufen werden,
dazu müssen Schnittstellenfunktionen in  \Cpp
geschrieben werden.

Man kann jedoch die Adressen von Objekten als void-Pointer
an C übergeben, und diese void-Pointer auch wieder
an \Cpp Code weitergeben. Diese void-Pointer müssen anschliessend in \Cpp in den
original Pointertypen zurück gecastet werden. Dieser Weg ist aber fehleranfaellig,
da der Programmierer dafür verantwortlich ist, dass der void-Pointer auch tatsaechlich
die Adresse des erwarteten Objektes enthält, ansonsten
ist mit undefiniertem Verhalten zu rechnen. 

Nun ein Beispiel zur Zusammenarbeit von C und \Cpp. (Wichtig, das ganze
muss mit dem \Cpp Linker und den \Cpp Bibliotheken gelinkt werden, d.h.
als \Cpp Projekt erstellt werden, sonst klappt es nicht).

EineKlasse.hpp
\begin{CCode}
#ifdef __cplusplus

/* Nur sichtbar wenn von C++ Compiler compiliert */
class EineKlasse {
        int x;
   public:
        EineKlasse(int NewX) : x(NewX) {};
   void Print(int a, float f);
};
/* Als gewoehnliche C Funktion codieren */ 
extern "C" void CallMethodPrint(void *Object, int a, float f);

#else

/* Fuer normalen C-Compiler ist nur dieser Teil sichtbar */ 
extern void CallMethodPrint(void *Object, int a, float f);
 
#endif
\end{CCode}


EineKlasse.cpp
\begin{CCode}
#include"EineKlasse.hpp"
#include"CModul.h"

void EineKlasse::Print(int a, float f)
{
   std::cout << "Eine Klasse sagt: " << a << " " << f << " " << x << std::endl;
}

void CallMethodPrint(void *Object, int a, float f) {
   /* void pointer in Zeiger auf Objekt casten Gefaehrlich, klappt nur */
   /* wenn void-pointer tatsaechlich auf ein objekt zeigt !!!          */
   ((EineKlasse *)Object)->Print(a, f);
}

int main(int argc, char *argv[]) 
{
   EineKlasse ek(3);
   
   /* Adresse von Objekt ek nach void * casten und C-Funktion aufrufen */
   TellCAboutIt((void *)(&ek));
}
\end{CCode}

CModul.h
\begin{CCode}
/* extern "C" nur fuer C++ Compiler sichtbar machen. Diese Variante */
/* ist besser und sicherer, da so der Prototyp fuer C und C++ nur   */
/* einmal geschrieben werden muss. (Weniger Fehleranfaellig)        */ 
#ifdef __cplusplus
extern "C" {
#endif

extern void TellCAboutIt(void *Object);

#ifdef __cplusplus
}
#endif
\end{CCode}


CModul.c
\begin{CCode}
#include"CModul.h"
#include"EineKlasse.hpp"

void TellCAboutIt(void *Object) 
{
   /* Ruft Schnittstellenfunktion in C++ auf */
   CallMethodPrint(Object, 42, 3.14);
}
\end{CCode}

\section{\Cpp-Checkliste}
Sobald eine Funktion virtuell ist, sollte auch der Destruktor virtuell sein.

Wenn eine Klasse einen Destruktor benötigt, sind im allgemeinen auch der
Copykonstruktor und der Zuweisungsoperator selbst zu schreiben, oder allenfalls
als privat zu deklarieren.

Für jede Klasse je eine eigene Deklarations- (Header-) und Implementationsdatei. (Ausnahme
Hilfsklassen)
